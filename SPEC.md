# haj 仕様

このドキュメントは **haj コアと、haj サブコマンドの間の契約** を定める。

サブコマンドの実装言語は問わない。この契約さえ満たせば、シェルでも Rust でも PHP でも
Python でも `haj` から等しく扱われる。逆に、コアはこの契約以外の方法でサブコマンドの
中身を知ろうとしない。

契約バージョン: **5**(`haj --version` はコアの版であって契約の版ではない)

- v5: 予約語 `secret` を追加(`haj secret` — §10.9)。`tree.<名前>.secret` は
  **宣言**(capability)へ — 注入は廃止し、秘密は常に pull(§10.8)。
  `haj secrets` は移行スタブに(§10.6)。`haj store` は裸の論理パスを取る(§10.10)
  - 追記(0.34.0): `haj secret list` / `check` と `haj config` に人手用の
    `--tree <インストール名>`(§10.9 / §10.8)。追加機能につき版は据え置き
  - 追記(0.35.0): `haj config --tree` が実効 env(全コマンドの `--haj-env`
    節連結、既定値込み)も出す — インスタンスの全景が1コマンドで閉じる
    (§10.8)。出力の拡張につき版は据え置き
  - 追記(0.36.0): `haj tree configure <名前>` とツリー規約 `config-init` —
    ツリーが `tree.*` の初期値を提案し、本人確認のうえユーザー設定へ追記する
    (§9.5 / §10.8 の入口)。任意の追加規約につき版は据え置き
- v4: 予約語 `store` とツリー専用ストア `store://`(§10.7)、ツリーごとの設定注入
  `tree.<名前>.env` / `.secret`(§10.8)、`haj store`(現 §10.10)を追加。
  コアが渡す環境変数に `HAJ_TREE`(§3.1)
  - 追記(0.32.0): store 設定を表形式に(`store.engine` / `store.prefix` →
    `store.tree.engine` / `store.tree.prefix`)。設定キーの変更のみで版は据え置き
- v3: ツリー名前空間と `expose`(§9.7)を追加。名前解決に「ツリー名」の層が入る
  - 追記(0.29.0): コアが渡す環境変数に `HAJ_START_DIR` / `HAJ_USER_CONFIG`(§3.1)。追加機能につき版は据え置き
  - 追記(0.28.0): 補完のファイル補完指示 `@files` / `@dirs`(§4.3, §6)。追加機能につき版は据え置き
- v2: 予約語 `run` とタスク(§9.6)を追加
- v1: 初版

## 1. 用語

| 語 | 意味 |
|---|---|
| コア | `haj` 実行ファイル本体。探索と exec だけを行う |
| サブコマンド | `haj <名前>` で呼ばれる実行可能ファイル |
| ツリー | `commands/` を含むディレクトリ。サブコマンドには `HAJ_ROOT` として渡される |
| タスク | プロジェクト局所の作業。`haj run <名前>` で呼ばれる(§9.6)。探索には乗らない |
| ストア | エンジン(金庫)の中に haj が確保する、**ツリー・インスタンス専用**の名前空間(§10.7)。`store://` で指す |
| 宣言 | `tree.<名前>.secret.*` — そのツリーが扱える秘密の**目録**(§10.8)。注入されず、`haj secret get` で引く |

## 2. 探索

コアはディスパッチ表を持たない。`haj <名前>` は、以下の順で **実行可能ファイル** を探し、
**最初に見つかったもの**を実行する。

| 順 | 探索先 | 用途 |
|---|---|---|
| 1 | カレントから上へ辿った各階層の `<dir>/.haj/commands/<名前>` | プロジェクト固有 |
| 2 | `~/.config/haj/commands/<名前>` | 個人用(XDG。§8.1) |
| 3 | `~/.local/share/haj/trees/<ツリー>/…/commands/<名前>` | インストール済みツリー(§9.5) |
| 4 | `$HAJ_COMMAND_PATH` の各ディレクトリ(`:` 区切り) | 全社/イメージ共通 |
| 5 | `$PATH` 上の `haj-<名前>` | git 方式の逃げ道 |

`$HAJ_COMMAND_PATH` の既定値は `/usr/local/lib/haj/commands`(設定ファイルの
`command_path` でも指定できる。§8)。

### 2.1 プロジェクト境界(`.haj` は壁である)

**1 の遡上は `/` までは行かない。`.haj` を持つディレクトリで止まる。**

素朴に `/` まで遡って見つかった `.haj/commands` を全部積むと、二つの困りごとが起きる。

1. **上流の野良 `.haj` が黙って効く。** 誰かが `~/repos/.haj/commands/setup` を置くと、
   その配下の**全リポジトリ**で `haj setup` が生えてしまう。置いた本人以外は気づけない。
2. **どのプロジェクトの `setup` が走ったのか分からない。** `setup` や `reset` は破壊的なので、
   これは事故になる。

そこで `.haj` を**プロジェクト境界**として扱う。カレントから遡って最初に見つけた `.haj` が
そのプロジェクトであり、**そこで探索を打ち切る**。

### 2.2 `.haj/config`

境界と名前は `.haj/config` で宣言する。**このファイルは無くてもよい**(既定値で成立する)。
`key = value` を並べただけの形式で、`#` から行末はコメント。行末の `\` は継続。

```
name = example-app
root = true
```

| 鍵 | 既定 | 意味 |
|---|---|---|
| `name` | `.haj` を含むディレクトリの名前 | 表示名。一覧の出自ラベルと `HAJ_PROJECT` に使う |
| `root` | `true` | `true` ならここで遡上を止める(境界)。`false` なら親の `.haj` も探しに行く |
| `alias.*` | (無し) | プロジェクト・エイリアス(§2.7) |
| `task.*` | (無し) | プロジェクト・タスクの1行宣言(§9.6) |

ファイル名も書式もユーザー設定と同じだが、**プロジェクト・スコープで効く鍵は上の
4種だけ(ホワイトリスト)**。`secrets.*` や `selfupgrade.*` をここから読むと、clone
したリポジトリに金庫や更新元の接続先を乗っ取られる。接続先・探索先を変える鍵は
ユーザー設定と環境変数からしか読まない(git の `~/.gitconfig` と `.git/config` の
関係と同型)。それ以外の鍵は黙って無視される。

**`root = false` はモノレポのサブプロジェクト用。** 親の共通コマンド(`mig` など)も
継承したい場合にだけ、明示的に壁を開ける。

既定が `true`(=止まる)なので、**知らないうちに上流のコマンドが効くことはない**。
継承は常にオプトイン。

### 2.3 これがサブセットの仕組み

「そのリポジトリに置いたコマンドだけが、そのリポジトリで生える」。
プロジェクトごとに使えるコマンドが違うことは、**有効化リストのような別管理ではなく、
探索結果そのもの**として表現される。無効化したければファイルを置かなければよい。

同名のコマンドがあれば、より手前(=より具体的)が勝つ。プロジェクトの `build` が
全社共通の `build` を上書きする。

### 2.4 素性は常に見えていなければならない

同じ `haj setup` がプロジェクトごとに違う挙動をする以上、**いまどれが効いているのかが
見えないことは、それ自体が欠陥**である。コアは次の3つでこれを担保する。

1. **`haj` は現在のプロジェクトを先頭に出す。**

   ```
    プロジェクト: example-app  (/home/kurari/repos/.../example-app)
   ```

2. **一覧はコマンドごとに出自を出す。**

   ```
    hajコマンド (haj help <名前> で詳細):
      bao-login  Vaultにログイン              [共通]
      setup      セットアップ                  [example-app]
   ```

   出自ラベルは `[<プロジェクト名>]` / `[個人]` / `[<ツリー名>]` / `[共通]` / `[PATH]`。

3. **`haj which --all <名前>` は隠れている定義まで並べる。**

   ```
   $ haj which --all setup
   * /home/kurari/repos/example-app/.haj/commands/setup  [example-app]
     /usr/local/lib/haj/commands/setup                          [共通]

   (* が実行されるもの。他は隠れている)
   ```

### 2.5 「実行可能ファイル」の判定

- 通常ファイルであること(ディレクトリは無視する)
- 実行ビットが立っていること(`mode & 0o111 != 0`)
- シンボリックリンクは辿る。リンク切れは無視する

`commands/` に README や `.gitkeep` を置いても、実行ビットが無ければコマンドにはならない。

### 2.6 名前の制約

以下はサブコマンド名として認められない(置かれても無視される)。

- 空文字列
- `.` または `-` で始まるもの
- `/` を含むもの
- **予約語**: `help`, `commands`, `which`, `config`, `completion`, `docs`, `env`, `exec`, `sh`, `run`, `selfupgrade`, `secret`, `secrets`, `store`, `tree`, `__complete`

予約語を弾くのは、`.haj/commands/help` を置かれるとコアのヘルプが奪われ、
「コマンド一覧を出す手段が無くなる」状態に陥りうるため。`exec` / `sh` はさらに強い理由で
奪われてはならない — 置き換えられると**展開済みシークレットを横取りする最短経路**になる(§9.2)。
`run` も同じ層にある — 奪われると「`haj run` はそのプロジェクトのタスクしか実行しない」
という §9.6 の保証が破れる。`store` も `exec` / `sh` の層 — 奪われると
`haj store put` の stdin(シークレットの**実体**)を横取りされる(§10.10)。
`secret` も同様 — 奪われると「`haj secret get` は宣言された参照しか解決しない」
という §10.9 の保証が破れる(偽の値の差し込み)。`secrets` は旧名の移行スタブ
(§10.6)として予約に残る — 探索に明け渡すと、旧名で書かれたスクリプトが
野良コマンドに落ちる。

### 2.7 エイリアス(git 方式)

`alias.<名前> = <語...>` と書くと、その名前が**語の並びに展開されてから**実行される。
残りの引数は後ろに繋がる。定義できる場所は3つで、**近いスコープが勝つ**
(コマンド探索と同じ並び):

1. プロジェクトの `.haj/config`(そのプロジェクトの中でだけ有効)
2. ユーザー設定 `~/.config/haj/config`
3. インストール済みツリーの `config`(§9.5 — エイリアス集を配れる)

```
# ~/.config/haj/config — どこでも使える個人の略記
alias.ie = -C ~/repos/example-app
```

```
# .haj/config — package.json の scripts に相当する「1行の委譲」
name = myapp
alias.test = exec docker compose exec app vendor/bin/phpunit --testdox
alias.test.desc = テストを流す(コンテナ内)
```

```sh
haj ie mig up    # → haj -C ~/repos/example-app mig up
haj ie help      # → そのプロジェクトのコマンド一覧
haj test         # (myapp の中で) → haj exec docker compose exec app ...
```

- 展開は名前の位置で**1回だけ**(再帰しない)。展開結果の先頭にグローバルフラグ
  (§3.2)があれば通常どおり解釈される。`-C` は展開より先に適用されるので、
  `haj -C ~/repos/myapp test` は移動先のプロジェクト・エイリアスが見える
- 優先順位: **予約語(組み込み) > エイリアス > ツリー名前空間(§9.7) > 探索**。
  `alias.help` のような予約語の名前は無視される
- **境界規則**: 1行で書けなくなったら `.haj/commands/` に昇格する。委譲は宣言、
  ロジックは実行ファイル(§11 — タスクランナーは作らない)。プロジェクトの作業なら
  同じ規則が `task.*` → `.haj/tasks/` に効く(§9.6)
- プロジェクト定義の露出は `.haj/commands/` と同じ(clone した中身が haj 経由で
  走る)。エイリアス固有の防壁は設けず、ツリーの信頼ゲート(Issue #1)で扱う
- 語の分割は空白区切り(引用符は解釈しない)
- **説明を書ける**: `alias.<名前>.desc = <一行説明>`。`haj help` の一覧と TAB 補完に出る
  (書かなければ展開そのものが出る。長い場合は切り詰められる)
- 長いエイリアスは**行末の `\` で継続**できる(§8.2):

```
alias.oci = --secret OCI_CLI_USER=vault://users/me/oci/user \
            --secret OCI_CLI_TENANCY=vault://users/me/oci/tenancy \
            --secret-file OCI_CLI_KEY_FILE=vault://users/me/oci/private_key \
            exec oci
```

- 素性は常に見える: `haj help` にエイリアスの節が出て、`haj which <名前>` は展開を
  表示し、TAB 補完にも名前が出る

`-C` の引数の先頭 `~` は HOME に展開される(§3.2)。設定ファイル由来の展開には
シェルがいないため、これが無いとエイリアスに `-C ~/...` が書けない。

## 3. 実行

コアは **`exec(2)` で自分自身を置き換える**。ラッパープロセスを残さない。したがって:

- サブコマンドの終了コードはそのまま呼び出し元へ伝わる
- シグナル(Ctrl-C など)はサブコマンドが直接受け取る
- `ps` に余計なプロセスが並ばない

### 3.1 コアが渡す環境変数

| 変数 | 値 |
|---|---|
| `HAJ_ROOT` | そのサブコマンドが属するツリー(`commands/` の親)の絶対パス |
| `HAJ_NAME` | 呼ばれた名前(`haj mig` なら `mig`) |
| `HAJ_PROJECT` | いま**操作対象になっている**プロジェクトの名前(§2.2) |
| `HAJ_PROJECT_DIR` | そのプロジェクトのディレクトリ(`.haj` を含む方) |
| `HAJ_START_DIR` | haj が起動された時点の cwd の絶対パス(`-C` **適用前**) |
| `HAJ_USER_CONFIG` | ユーザー設定ファイルのパス(§8。XDG 解決後、存在しなくても渡す) |
| `HAJ_TREE` | **インストール済みツリー**(§9.5)由来のコマンドにだけ: そのツリーの**インストール名**。それ以外では未設定 |

`HAJ_ROOT` は**共通ライブラリを解決するためにある**。サブコマンドは

```sh
. "$HAJ_ROOT/lib/common.sh"
```

と書けばよい。プロジェクト固有のコマンドは**そのプロジェクトの lib** を読むことになり、
全社共通の lib と混線しない。

`HAJ_PROJECT` は**破壊的なコマンドが対象を名乗るためにある**。`setup` や `reset` は、
何に対して実行しているのかを出力に出すべきである。

```sh
echo "==> ${HAJ_PROJECT}: セットアップします (${HAJ_PROJECT_DIR})"
```

**`HAJ_ROOT` と `HAJ_PROJECT` は別物である。** 前者は「そのコマンドがどこから来たか」、
後者は「いまどこに対して実行しているか」。全社共通の `mig` をプロジェクトの中で叩けば、
`HAJ_ROOT=/usr/local/lib/haj` だが `HAJ_PROJECT=example-app` になる。
`root = false` の入れ子でも両者はずれる。

`$PATH` 上の `haj-<名前>` で見つかった場合はツリーが無いので、`HAJ_ROOT` は
**設定されない**(呼び出し元の環境に残っていても明示的に削除される)。同様に、
プロジェクトの外で実行した場合は `HAJ_PROJECT` も設定されない。

`HAJ_START_DIR` は**「ユーザーが元居た場所」を対象にするためにある**。`-C` は
その場で cwd を移すため、サブコマンドから見える cwd は移動後になる。移動前の
場所はこの変数でしか分からない(`-C` を使っていなければ cwd と一致する)。

`HAJ_USER_CONFIG` は**ユーザー設定の場所の解決をコアに一元化するためにある**。
サブコマンドが同じファイルに設定を相乗りする・参照するとき、XDG の解決ロジックを
複製しなくてよい。ファイルが存在するかは読み手が判断する(「読みに行く場所」を
渡すのであって、存在の保証ではない)。

`HAJ_TREE` は**同じツリーの多重インストールをインスタンスとして区別するためにある**。
同じリポジトリを `haj tree install <URL> --name a` と `--name b` で2つ入れて
アカウントを使い分けるとき、ツリーのコードは自分がどの名前で入れられたかを知らない —
ローカル状態(取得したトークン等)の置き場をコードに固定で書くと、2つのインスタンスが
**同じファイルを奪い合う**。状態パスにこの変数を織り込めば衝突しない:

```sh
state_dir="${XDG_STATE_HOME:-$HOME/.local/state}/myext/${HAJ_TREE:-default}"
```

- 値は**インストール名**(`trees/<名前>` のディレクトリ名。install 時に `--name` で
  選んだもので、名前空間の名(§9.7)と同じ)。`HAJ_PROJECT`(名前)/
  `HAJ_PROJECT_DIR`(パス)の対と同じ流儀で、こちらの対は `HAJ_TREE`(名前)/
  `HAJ_ROOT`(パス)
- **呼び出しの形に依存しない**: 名前空間形(`haj <ツリー名> <名前>`)でも素の探索
  (flat ツリー — §2 の順 3)でも、値はインストール先のパスから決まる同じもの。
  個人と `--global`(§9.5)のどちらのスコープでも同様
- **インストール済みツリー以外では未設定**(呼び出し元の環境に残っていても明示的に
  削除される)。プロジェクト(`.haj/commands` / `.haj/tasks`)・個人
  (`~/.config/haj/commands`)・共通(`$HAJ_COMMAND_PATH`)・`$PATH` の `haj-<名前>`
  は「インストールされたインスタンス」ではないので、この概念が無い
- 注入は他の変数と同じく規約フック(§4)の呼び出しにも行われる

これらの変数は**コアが注入する実行時情報**であり、`--haj-env`(§4.4)で
サブコマンドが**申告する設定値**とは別物である。前者は haj の都合(どこから
起動されたか)、後者はコマンドの都合(何を読むか)。注入は規約フック(§4)の
呼び出しにも行われる。

### 3.2 グローバルフラグ

haj 自身のフラグは**サブコマンド名の前にだけ**書ける(git 方式)。名前以降は §11 の
とおり無解釈で素通しなので、サブコマンドの引数と衝突しない。

| フラグ | 意味 |
|---|---|
| `-C <ディレクトリ>` | そのディレクトリを起点に実行する(下記)。先頭の `~` は HOME に展開 |
| `--secret` / `--env-file` / `--secret-file` | シークレットの明示的な受け渡し(§10.2) |

**`-C` は git と同じ意味論。** haj は起動直後に chdir するので、探索
(`.haj/commands` の遡上・プロジェクト境界)、`HAJ_PROJECT` / `HAJ_PROJECT_DIR`、
サブコマンド(`exec` / `sh` を含む)の cwd がすべて移動先を起点になる。

```sh
haj -C ~/repos/example-app mig up
haj -C ~/repos/example-app help    # そのプロジェクトのコマンド一覧
```

- 複数指定できる。2個目以降の相対パスは**直前の `-C` からの相対**(git と同一):
  `haj -C /a -C b` → `/a/b`
- 移動できなければ実行前に exit 1(fail-fast)
- `--env` の相対パスは移動**後**に解決される(フラグの適用は exec の直前のため)

`haj help` はグローバルフラグの一覧も出す(一覧が実態と一致する、という約束の一部)。

### 3.3 コアが返す終了コード

| コード | 意味 |
|---|---|
| サブコマンドの値 | 正常に exec できた場合(コアは何も足さない) |
| `127` | 未知のコマンド(シェルの command not found に合わせる) |
| `126` | 見つかったが exec に失敗した |
| `1` | コア自身の使い方の誤り(`haj which` に引数が無い等) |

## 4. コマンド規約

サブコマンドは、以下の引数で呼ばれたときの応答を実装する。

### 4.1 `--haj-describe` (必須)

**一行**の説明を stdout に出し、`0` で終了する。

```sh
$ haj-mig --haj-describe
DBマイグレーション (status/up/down/create/edit)
```

`haj help` のコマンド一覧はこれを全コマンドに聞いて**自動生成**される。
複数行返しても 1 行目だけが使われる。

これを実装しなくても**コマンドとしては動く**(一覧に説明が出ないだけ)。ただし
一覧の価値が落ちるので必須とする。

### 4.2 `--haj-help` (任意)

複数行の詳しい使い方を stdout に出し、`0` で終了する。`haj help <名前>` で表示される。

実装していない場合は `--haj-describe` の出力にフォールバックする。

### 4.3 `--haj-complete <語...>` (任意)

補完候補を**改行区切り**で stdout に出し、`0` で終了する。

`<語...>` は **そのサブコマンド以降に入力済みの語**。カーソル位置の語は含まない。

**語数の規則(最も間違えやすい)**

| ユーザーの入力 | 渡される語 | 返すべきもの |
|---|---|---|
| `haj mig ⇥` | (なし) | 1 語目の候補 |
| `haj mig up ⇥` | `up` | 2 語目の候補 |
| `haj mig create foo ⇥` | `create` `foo` | 3 語目の候補 |

つまり **`$#` が 0 なら 1 語目を補完する**。`[ $# -le 1 ]` と書くと 1 つずれる。

候補を列挙できない箇所(自由入力)では、丸括弧で囲んだ 1 行を返す。シェル補完は
これを候補ではなく**説明文**として表示する。

```
(新しいマイグレーションの名前)
```

**ファイル補完の指示(`@files` / `@dirs`)**

パスを取る箇所では、**1 行目**に指示を返すとシェルがファイルシステム補完を行う
(ファイルシステムの補完はシェルにしかできない。コアは素通しで届ける — §6)。

```
@files                     → ファイルとディレクトリ
@files<TAB><glob>...       → glob に合うファイルだけ(ディレクトリは降りるために常に出る)
@dirs                      → ディレクトリのみ
```

- glob は **タブ区切りで複数**書ける。書式は sh のパターン(`*.mp4` など。
  zsh 拡張や空白を含む glob は書けない)。ファイル名(パスの最後の要素)に対して照合される
- 指示行の**後に続く行**は、通常の候補としてファイルと併せて提示される
  (丸括弧 1 行の説明もこれまでどおり有効)

```sh
# haj transcode <src> <out_dir> の補完
--haj-complete) shift
  case $# in
    0) printf '@files\t*.mp4\t*.mkv\t*.mov\n' ;;   # 1語目: 入力ファイル
    1) echo '@dirs' ;;                             # 2語目: 出力先ディレクトリ
  esac
  exit 0 ;;
```

1 行目の `@` 始まりは指示として予約する。`@` で始まる普通の候補を返したい場合は、
1 行目に置かないこと。

### 4.4 `--haj-env` (任意)

そのコマンドが**読む環境変数**を `KEY=value` で stdout に出し、`0` で終了する。
`haj env <名前>` がこれを中継する。

```console
$ haj env metrics
# 対象DBの接続先
DB_HOST=db.staging.internal
LONG_TX_SEC=60
```

- 値には既定値か、いま効いている値を出す(`${DB_HOST:-db.staging.internal}` のように
  現在の環境を反映してよい。フックは呼び出し時の環境で走る)
- `#` から行末はコメント。**出力はそのまま `--env-file` に渡せる形式**にする:

```sh
haj env metrics > env.txt    # 雛形を出す
vi env.txt                   # 値を書き換える(シークレット参照も書ける)
haj --env-file env.txt metrics
```

「どの環境変数を読むのか」はコマンドの中身の知識なので、コアは聞くだけ
(describe と同じ思想)。ただし規約を知らないコマンドはこのフラグを無視して
本処理の出力を返してくるため、コアは**形を検証する**: 空行と `#` コメント以外の
行がすべて `KEY=value` でなければ、未実装として扱う(「そのまま `--env-file` に
渡せる」という約束を壊れた出力から守る)。

### 4.5 規約フックの制約

- **stdin は `/dev/null`** に接続される。フックで入力を待ってはならない
- **stderr は捨てられる**
- **タイムアウトは既定 2 秒**(`HAJ_HOOK_TIMEOUT_MS` で変更可)。超えると `SIGKILL` され、
  そのコマンドは「説明なし/候補なし」として扱われる。他のコマンドは巻き添えにならない
- **副作用を持ってはならない。** フックは `haj help` と TAB のたびに呼ばれる

### 4.6 実装の型

規約フックは**共通ライブラリを読む前に**処理すること。説明文を 1 行返すためだけに
DB の環境変数を読み込むのは無駄で、TAB の体感速度に直結する。

```sh
#!/bin/bash
set -euo pipefail

case "${1:-}" in
  --haj-describe) echo "一行説明"; exit 0 ;;
  --haj-help)     cat <<'EOF'
詳しい使い方
EOF
    exit 0 ;;
  --haj-complete)
    shift
    if [ $# -eq 0 ]; then printf '%s\n' up down; else printf '%s\n' v0 v2; fi
    exit 0 ;;
esac

# ここから先が本体。重い初期化はフックの後に置く。
. "${HAJ_ROOT:?hajルーター経由で実行してください}/lib/common.sh"
...
```

## 5. ヘルプの構成

`haj help` の出力は次の 3 つを連結したもの。

```
help.header      ツリー直下のファイル。あれば先頭に出す(任意)
   ↓
docsへの誘導     「ドキュメント: haj docs」の1行(固定)
   ↓
コマンド一覧      --haj-describe を全コマンドに聞いて自動生成
   ↓
タスク一覧        プロジェクトの中でだけ出る。task.* と .haj/tasks/ から自動生成(§9.6)
   ↓
名前空間ツリー     expose = namespace のツリーの入口(§9.7)。1行に畳む
   ↓
help.footer      ツリー直下のファイル。あれば末尾に出す(任意)
```

docs への誘導を冒頭に出すのは、ドキュメントの入口が「haj自身」の一覧の中に
埋もれないようにするため(§9.3)。

`help.header` / `help.footer` は**探索順で最初に見つかったツリー**のものが使われる。
プロジェクトが自分の header を置けば、全社共通の header を上書きできる。

ここに置くのは「コマンド以外の案内」だけ(URL、環境変数の上書き方法、ファイルの
置き場所など)。**コマンド一覧は自動生成されるので、手で書いてはならない。**

## 6. 補完プロトコル

シェル補完スクリプトは候補を一切持たない。コアに聞くだけ。

```
haj __complete [<語...>]           → 入力済みの語(グローバルフラグ込み)を素通しで渡す
```

コアは本体と同じ規則で先頭のグローバルフラグを読み飛ばし(`-C` は実際に移動する —
移動先のコマンドを補完するため)、残りが 0 語なら **"名前\t説明"** の一覧
(組み込みとエイリアスも含む)を、1 語以上なら先頭をコマンド名として
**その `--haj-complete` へ転送**する。値が未入力のフラグで終わっていたら候補は返さない
(**フラグの値の補完だけはシェル側の仕事**: `--env-file` はファイル、`-C` は
ディレクトリ、`--secret*` の `名前=` の右辺はファイル。ファイルシステムの補完は
シェルにしかできない)。

エイリアス(§2.7)は**展開してから**転送する。`alias.pj = -C ~/repos/x` なら
`haj pj <TAB>` は移動先のプロジェクトのコマンドを補完する。

**タスク(§9.6)**: 1語目が `run` なら、残りが 0 語のとき **"名前\t説明"** のタスク一覧を、
1 語以上のとき先頭をタスク名として**その `--haj-complete` へ転送**する。config のタスク
(`task.*`)はエイリアスと同じく展開してから扱う(`exec` に解決されれば `@delegate`)。

**ツリー名前空間(§9.7)**: 1語目がインストール済みツリーの名前なら、残りが 0 語のとき
そのツリーのコマンド一覧("名前\t説明")を、1 語以上のとき先頭をコマンド名として
その `--haj-complete` へ転送する。0 語の一覧(コマンド名の位置)にはツリー名自体も出る。

**委譲(`@delegate`)**: 実効コマンドが `exec`(haj の外のコマンド)に解決されたとき、
コアは候補を作れない。代わりに1行目に指示を返す。

```
@delegate<TAB><プログラム><TAB><引数...>
```

シェル補完はこれを見たら、**そのプログラム自身の補完に委譲する**(zsh なら `words` を
組み替えて `_normal`。sudo / env などのラッパーと同じ定石)。これで
`haj exec kubectl <TAB>` や、`exec` を含むエイリアス(`haj oci <TAB>`)でも、
そのコマンドの補完がそのまま効く。

**ファイル補完(`@files` / `@dirs`)**: サブコマンドの `--haj-complete` が 1 行目に
返す指示(§4.3)。コアは**素通しで**シェルへ届け、シェル補完スクリプトがファイル/
ディレクトリ補完(zsh なら `_files` / `_directories`)を行う。指示行の後に続く行は
通常の候補として併せて提示する。

つまり 1 行目が `@` で始まる出力は**シェルへの指示**であり、指示は
`@delegate`(コアが作る)と `@files` / `@dirs`(サブコマンドが作る)の 3 つ。
シェル補完スクリプトは未知の `@` 指示を候補として表示してはならない
(将来の指示の追加に黙って耐えるため)。

したがって**サブコマンドを追加しても補完スクリプトを更新する必要がない**。
これが `--haj-complete` を規約に含める理由。

## 7. 説明文のキャッシュ

`haj help` と TAB 補完はコマンドの数だけサブプロセスを起動する。bash のサブコマンド
1 本あたり実測で 9ms 程度かかるため、20 本あれば TAB のたびに約 200ms 待たされる。

コアは `--haj-describe` の結果を **(パス, 更新時刻, サイズ)** を鍵にキャッシュする。

- 置き場所: `$XDG_CACHE_HOME/haj/describe.tsv`(既定 `~/.cache/haj/describe.tsv`)
- ファイルを触っていないコマンドは二度と起動されない
- `HAJ_NO_CACHE=1` で無効化できる
- 書き込みは一時ファイル + `rename` で行う(補完は同時に何本も走りうるため)

キャッシュは**壊れても消えても正しく動く**(聞き直すだけ)。

## 8. 設定

### 8.1 場所(XDG に従う)

**git と同じ形**。リポジトリ側は `.haj/`(git の `.git/`)、ユーザー側は
`~/.config/haj/`(git の `~/.config/git/config`)。

| 何 | 場所 |
|---|---|
| ユーザー設定 | `$XDG_CONFIG_HOME/haj/config`(既定 `~/.config/haj/config`) |
| 個人用コマンド | `$XDG_CONFIG_HOME/haj/commands/` |
| 説明文キャッシュ | `$XDG_CACHE_HOME/haj/describe.tsv`(既定 `~/.cache/haj/`) |
| プロジェクト設定 | `<リポジトリ>/.haj/config` |

ユーザーとプロジェクトのツリー構成は**ファイル名まで対称**である
(`config` + `commands/` + `docs/`)。覚える構造は1つでよい。
唯一の例外は `tasks/`(§9.6)— タスクはプロジェクト局所の概念なので、
プロジェクト・スコープにしか存在しない。
旧 `~/.haj/`(0.3.0 までの置き場)はもう読まない。

### 8.2 形式

**`.haj/config` と同じ `key = value`。** `#` から行末はコメント。
**行末の `\` は継続**(シェルや git config と同じ)。継続行は空白1つで繋がる。

設定ファイルの形式が2つあると「どっちがどっちだったか」を覚える羽目になる。
入れ子が要るような項目は無く、信頼済みツリーの一覧のような**列**は direnv 方式で
別ファイルにすればよい。これで依存クレートをゼロに保てる(YAML/TOML はパーサが要る)。

鍵は **git と同じドット記法**で名前空間を持つ(`selfupgrade.token` など)。パーサに
とってドットはただの文字なので、入れ子の構文(TOML 等)は要らない。ドット無しはコア。

```
# ~/.config/haj/config
selfupgrade.token = vault://secret/data/haj/token
secrets.vault_cmd = bao

hook_timeout_ms = 2000
```

設定できる鍵と既定値は `haj config --init` が**すべて**雛形として出す(全行コメント
なので、そのまま置いても挙動は変わらない。変えたい行だけコメントを外す):

```sh
haj config --init > ~/.config/haj/config
```

### 8.3 優先順位

**環境変数 > 設定ファイル > 既定値。**

この3段が見えないと「なぜ効かないのか」を調べる手段が無くなるので、
**`haj config` は実効値と一緒に必ず出所を出す**(`haj which` が探索順を見せるのと
同じ理由)。`token` は値を出さず、設定されているかと出所だけを示す。

| 設定ファイルの鍵 | 環境変数 | 既定 | 意味 |
|---|---|---|---|
| `command_path` | `HAJ_COMMAND_PATH` | `/usr/local/lib/haj/commands` | システム共通のコマンド置き場(`:` 区切り) |
| `hook_timeout_ms` | `HAJ_HOOK_TIMEOUT_MS` | `2000` | 規約フックのタイムアウト |
| `docs.fzf_cmd` | `HAJ_DOCS_FZF_CMD` | `fzf` | docs の選択UIに使うコマンド(語分割。先頭がバイナリ)(§9.3) |
| `docs.fzf_args` | `HAJ_DOCS_FZF_ARGS` | (無し) | 選択UIへ**追加**で渡す引数。haj の既定の後ろに付く(fzf は後勝ちなので上書きできる)(§9.3) |
| `docs.preview_cmd` | `HAJ_DOCS_PREVIEW_CMD` | (無し) | プレビューのフィルタ。markdown を stdin で受ける(例: `glow -`)(§9.3) |
| `docs.pager` | `HAJ_DOCS_PAGER` | `$PAGER`、無ければ `less` | Enter で開くビューア(§9.3) |
| `secrets.op_cmd` | `HAJ_OP_CMD` | `op` | op 参照の解決に使う CLI(§10) |
| `secrets.vault_cmd` | `HAJ_VAULT_CMD` | `vault` | vault 参照の解決に使う CLI(OpenBao なら `bao`)(§10) |
| `secrets.vault_addr` | `VAULT_ADDR` | (無し) | vault サーバ。環境の `VAULT_ADDR` / `BAO_ADDR` が優先(§10.4) |
| `secrets.vault_login` | `HAJ_VAULT_LOGIN` | `off` | 未ログイン時に自動実行する `login` の引数(§10.4)。`off` で無効 |
| `store.tree.engine` | `HAJ_STORE_TREE_ENGINE` | `vault` | ストア `tree` のエンジン(§10.7)。v1 は `vault` のみ |
| `store.tree.prefix` | `HAJ_STORE_TREE_PREFIX` | `secret/data/users/<ユーザー名>` | ストア `tree` の物理プレフィックス(§10.7)。書式は物理参照のパスと同じ |
| `tree.<名前>.env.KEY` | (無し) | (無し) | ツリーごとの設定注入: 平文(§10.8)。**設定ファイル専用**(下記) |
| `tree.<名前>.secret.KEY` | (無し) | (無し) | ツリーの秘密の**宣言**(§10.8)。注入されない — `haj secret get` で引く。**設定ファイル専用**(下記) |
| `selfupgrade.github` | `HAJ_GITHUB` | `AvapCoLtd/haj` | 取得元の GitHub リポジトリ(public。認証不要)(§9.1) |
| `selfupgrade.target` | `HAJ_TARGET` | `x86_64-unknown-linux-musl` | 取得するビルドのターゲット |
| `selfupgrade.gitlab` | `HAJ_GITLAB` | (無し) | private な取得元を使うとき(§9.1) |
| `selfupgrade.project_id` | `HAJ_PROJECT_ID` | (無し) | 同上。GitLab のプロジェクト ID |
| `selfupgrade.token` | `HAJ_TOKEN` | (無し) | 同上。**シークレット参照を書ける**(§8.4) |

環境変数だけのもの(設定ファイルに書けない):

| 変数 | 意味 |
|---|---|
| `HAJ_NO_CACHE` | `1` で説明文キャッシュを無効化(デバッグ用) |
| `XDG_CONFIG_HOME` / `XDG_CACHE_HOME` | 置き場所そのものを決めるので、設定ファイルには書けない |

逆に**設定ファイルだけのもの**(対応する環境変数が無い): `tree.<名前>.env.KEY` /
`tree.<名前>.secret.KEY`(§10.8)。動的な鍵名(インストール名と変数名)に環境変数の
写像を発明しない — その場かぎりの上書きは、環境変数そのもの(シェル環境が tree設定に
勝つ — §10.8)と `--secret`(§10.2)で既にできる。

### 8.4 `selfupgrade.token` にはシークレット参照を書ける

```
# ~/.config/haj/config
selfupgrade.token = vault://secret/data/haj/token
```

平文の PAT をディスクに置かずに、素の `haj selfupgrade` が動く。

- 展開は **token を使うとき**(selfupgrade の認証時)に行う。読むだけの `haj config` では
  解決しない(参照は参照のまま表示する。参照は秘密ではない)
- 判定は値全体規則(§10.3)。参照でなければ従来どおり平文として使う。`HAJ_TOKEN`
  環境変数に参照が入っている場合も同様に展開する
- **ゲートは不要。** `~/.config/haj/config` は本人にしか書けない
  ファイルであり、参照を書いたこと自体が同意(`--secret` フラグと同じ層)
- 解決に失敗したら selfupgrade を進めない(fail-fast)
- 展開されるのは `selfupgrade.token` **だけ**。`secrets.op_cmd` / `secrets.vault_cmd` など
  などリゾルバ自身の設定は展開しない(参照の解決に参照が要る、という再帰を作らない)

## 9. コア組み込みのコマンド

| コマンド | 用途 |
|---|---|
| `haj` / `haj help` | ヘルプ(header + 自動生成の一覧 + footer) |
| `haj help <名前>` | そのコマンドの `--haj-help` |
| `haj commands` | `名前\tパス\t出自\t説明` を機械可読で列挙 |
| `haj which [--all] <名前>` | 探索で勝っている実行ファイルのパス(`--all` で隠れているものも) |
| `haj config [--init \| --tree <インストール名>]` | 設定の実効値と出所。`--init` は雛形、`--tree` はそのインスタンスに効く設定(§10.8) |
| `haj selfupgrade` | コア自身の更新(§9.1) |
| `haj secret <動詞> ...` | 宣言された秘密を引く: get / list / check(§10.9)。check は受け渡しの事前確認も(§10.6) |
| `haj store <動詞> ...` | 自ツリーのストアの読み書きと認証: get / put / login / status(§10.10) |
| `haj env <名前>` | そのコマンドが読む環境変数を `KEY=value` で出す(`--haj-env` の中継。§4.4) |
| `haj run [<名前>] [引数...]` | プロジェクトのタスクを実行する。引数なしは一覧(§9.6) |
| `haj exec [--] <プログラム> [引数...]` | PATH のコマンドにシークレットを注入して実行(§9.2) |
| `haj sh '<コマンド>' [引数...]` | `exec sh -c` の省略形(§9.2) |
| `haj docs [<トピック>]` | 端末で読めるドキュメント(§9.3) |
| `haj completion <シェル>` | 補完スクリプトを出す(§9.4) |
| `haj --version` | コアの版 |
| `haj __complete ...` | 補完プロトコル(人間向けではない) |

### 9.0 組み込みは常に見えていなければならない

**組み込みコマンドは探索の対象ではないが、`haj help` の一覧にも補完にも `haj commands` にも
出す。** どこにいても使えるものを一覧から漏らせば、「`haj help` の一覧が実態と一致する」
という haj の約束そのものが嘘になる。

一覧では、プロジェクトのコマンドとは節を分けて出す(性質が違うため)。

```
 hajコマンド (haj help <名前> で詳細):
   mig          DBマイグレーション   [example-app]
   bao-login    Vaultにログイン      [共通]

 haj自身 (どのプロジェクトでも使える):
   help         使い方を表示する (haj help <名前> で個別)
   commands     コマンド一覧を機械可読で出す
   which        どの定義が効いているかを見る (--all で隠れているものも)
   selfupgrade  haj自身を更新する
   secret       宣言された秘密を引く (get / list / check)
   store        自ツリーのストアを読み書きする (get / put / login / status)
   env          コマンドが読む環境変数を key=value で出す (--env-file にそのまま渡せる)
   run          プロジェクトのタスクを実行する (引数なしで一覧)
   exec         PATHのコマンドにシークレットを注入して実行する
   sh           シェルの1行をシークレットを注入して実行する (exec sh -c の省略形)
   docs         ドキュメントを読む (コマンドの作り方・仕様・ツリーの文書)
   completion   シェル補完のスクリプトを出す
```

`haj commands` では出自ラベル `[haj]`、パスの代わりに `(組み込み)` を出す。

組み込みも `haj help <名前>` に答え、補完候補を返す(`haj selfupgrade ⇥` → `--check`、
`haj which ⇥` → `--all` とコマンド名)。実装がプロセス起動ではないだけで、**サブコマンドと
同じ顔をしている**。

### 9.1 `haj selfupgrade`

コア自身をリリース版で置き換える。チェックアウト無しで「入っているhajを更新する」
ためのもの。

```
haj selfupgrade [<版>] [--check]
```

**取得元は2つ。既定は GitHub Releases(認証不要)。**

| 取得元 | 使うとき | 認証 |
|---|---|---|
| GitHub Releases(既定) | 設定 `selfupgrade.github`(既定 `AvapCoLtd/haj`) | **不要**(public) |
| GitLab の Package Registry | `selfupgrade.gitlab` と `selfupgrade.project_id` を設定したとき | `selfupgrade.token` が要る |

private な社内配布に切り替えるなら、設定ファイルに書く:

```
selfupgrade.gitlab     = https://gitlab.example.com
selfupgrade.project_id = 123
selfupgrade.token      = vault://<マウント>/<パス>/token
```

- 引数なし: 最新リリースを調べ、今の版と**違えば**置き換える。同じなら何もしない
- `<版>` 指定: その版を**無条件に**入れる。同版の再インストールもダウングレードも
  これで表現する(`--force` は設けない)
- `--check`: 調べるだけ。終了コードで返す — `0` 最新 / `1` 更新あり / `2` 調べられなかった
- 版の比較は**文字列の一致のみ**。semver は解釈しない(違えば更新とみなす)

**取得と検証の手順**

1. 版が未指定なら releases API から最新の `tag_name` を取る(先頭の `v` は落とす)
2. `haj-<ターゲット>.tar.gz` を取得元からダウンロードする
3. `.sha256` が取れたら `sha256sum -c` で照合する。**不一致は中止**。
   `.sha256` 自体が無ければ照合は省略する(install.sh と同じ)
4. 展開した新バイナリを `--version` で起動し、出力が `haj ` で始まることを確認する。
   壊れたバイナリで自分を潰さないための健全性チェック
5. 現バイナリと**同じディレクトリ**に一時ファイル(mode 0755)を書き、`rename(2)` で
   置き換える。同一ファイルシステム内なので原子的で、実行中のプロセスに影響しない
6. 置き場所に書けなければ、sudo での再実行を**提案して**終了する。コアは昇格しない

**依存ゼロとの折り合い**: stdlib に HTTP は無いが crate は増やさない。取得・展開・照合は
`curl` / `tar` / `sha256sum` を子プロセスとして呼ぶ(exec がコアの本業、という設計に沿う)。
`curl` が無い環境ではその旨を伝えて終了する。

**読む環境変数**(install.sh と共通)

| 変数 | 既定 | 意味 |
|---|---|---|
| `HAJ_GITLAB` | `https://gitlab.example.com` | GitLabインスタンス |
| `HAJ_PROJECT_ID` | `788` | hajのプロジェクトID |
| `HAJ_TARGET` | `x86_64-unknown-linux-musl` | 取得するビルドのターゲット |
| `HAJ_TOKEN` | (必須) | 取得に使うトークン。`CI_JOB_TOKEN` と同値なら JOB-TOKEN ヘッダを使う |

### 9.2 `haj exec`

探索を**通さず**、PATH のコマンド(`/` を含めばそのパス)に**シークレットの注入だけ
して**実行する。`op run` / `doppler run` が占めている場所。「注入は欲しいが、haj の
コマンドにするほどではない」一回きりの実行のためにある。

```sh
haj --secret DB_HOST=vault://secret/data/db/host exec sh -c 'mysql -h $DB_HOST'
```

- 展開の規則(§10)はサブコマンド実行と完全に同じ。受け渡しフラグも同様に乗る
- **シェルの変数展開が要るなら、明示的に `sh -c` を書くか、省略形 `haj sh` を使う。**
  コアは文字列をシェルに包まない(§11 — コアは引数を解釈しない)。`$VAR` を使わないなら
  `haj exec mysql -h db.example` のようにそのまま並べればよい
- `HAJ_ROOT` / `HAJ_NAME` / `HAJ_TREE` は**渡さない**(明示的に削除する)。haj の外の世界の
  コマンドに、hajサブコマンドの顔をさせない
- `HAJ_PROJECT` / `HAJ_PROJECT_DIR` は**渡す**(§3.1 と同じ規則)。これは「サブコマンド
  であること」ではなく「どこに対して実行しているか」の情報で、プロジェクト・エイリアス
  が `sh --` へ委譲したときに対象を名乗れないのは §2.4 に反する
- 見つからなければ `127`、exec に失敗したら `126`(§3.3 と同じ)
- `exec` は予約語(§2.6)。`.haj/commands/exec` で置き換えられると、`--secret` で
  展開したシークレットを横取りする最短経路になるため、**探索より常に優先**する

**`haj sh '<コマンド>' [引数...]`** は `haj exec sh -c '<コマンド>' ...` の省略形。
追加の引数はシェルの位置パラメータ(`$1` から)として渡る。`sh` も同じ理由で予約語。

```sh
haj --secret MYSQL_HOST=vault://secret/data/db/host sh 'mysql -h $MYSQL_HOST'
haj sh 'echo $1-$2' one two    # → one-two
haj sh -- ls -la               # `--` の後は語を空白で繋いで1行にする(ssh 方式)
```

**先頭の `--` は両方が受け付ける。** `haj exec -- <プログラム> [引数...]` は読み飛ばすだけ
(`op run --` / `kubectl exec --` の指癖と互換)。`haj sh -- <語...>` は以降の語を空白で
繋いで1行のスクリプトにする。引用が要る引数を含むなら `haj sh '<コマンド>'` の
1文字列形式で書くこと。

### 9.3 `haj docs`

端末で読めるドキュメント。**コマンドと同じ思想で探索に乗る。**

```
haj docs                     # トピック一覧(出自つき)
haj docs writing-commands    # コマンドの作り方(haj同梱)
haj docs spec                # この仕様の全文(haj同梱)
```

- ツリーは `commands/` と並んで **`docs/`** を持てる: `<ツリー>/docs/<トピック>.md`。
  探索順・境界・先勝ちはコマンドと同一で、一覧には出自ラベル(§2.4)が付く。
  プロジェクトの手順書(onboarding 等)がそのプロジェクトの中でだけ生える
- 一覧の一行説明は**ファイル先頭の見出し行**(`# タイトル`)から取る。
  説明を別ファイルに書かせない(規約フックの docs 版)
- haj 同梱のドキュメント(`writing-commands` / `secrets` / `trees` / `spec`)は
  バイナリに埋め込まれていて、どこでも読める。出自は `[haj]`。
  **ツリー側の同名トピックが勝つ**
  (コマンドの予約語と違い、奪われても実害が無い。埋め込みは最後のフォールバック)
- 出力は素の markdown を stdout へ。ページャは使う側のパイプに任せる:
  `haj docs spec | less`
- **引数なしで stdout が端末(TTY)かつ `fzf` が PATH にあるときだけ、一覧が fzf の
  選択UIになる**(プレビューに本文、Enter で `${PAGER:-less}` 表示、Esc で何もせず終了)。
  fzf が無い・TTY でない(パイプやリダイレクト)ときは従来どおり一覧を印字するだけ
  なので、スクリプトからの利用は変わらない。fzf / ページャは CLI への委譲
  (op / bao / git と同じ流儀)で、依存クレートは増やさない
- 選択UIは設定で差し替え・味付けできる(§8.3)。`docs.fzf_cmd` は UI コマンドそのもの、
  `docs.fzf_args` は追加引数(`--preview-window=up,90%,wrap` などで「fzf のまま読み切る」
  構えにできる)、`docs.preview_cmd` はプレビューを markdown レンダラへ通すフィルタ
  (本文を stdin で受ける。例: `glow -`)、`docs.pager` は Enter のビューア。
  値はどれも空白で語分割するだけ(引用符は解釈しない — `secrets.vault_login` と同じ流儀)

### 9.4 `haj completion`

補完スクリプトを標準出力に出す。**補完はファイルとして配らない** — バイナリ1本で
完結する(gh / kubectl / rustup と同じ流儀)。

```sh
# ~/.zshrc
eval "$(haj completion zsh)"

# ~/.bashrc
eval "$(haj completion bash)"
```

スクリプト自体は候補を持たない(§6)。`haj __complete` に聞くだけなので、コマンドを
足しても補完は自動で追従する — プロジェクト固有のコマンドもそのまま候補に出る。


### 9.5 `haj tree` — 共有ツリーの配布

コマンドのツリーを git リポジトリで配る。**パッケージマネージャではない** —
clone したディレクトリが探索の対象(§2 の順 3)になるだけで、コアの原理
(探索と exec)は変わらない。

```sh
haj tree install <gitのURL>[@<ref>] [--name <名前>] [--global]
haj tree update [<名前>]       # 省略で全部
haj tree list
haj tree remove <名前>
haj tree configure <名前>      # ツリーの初期値提案をユーザー設定へ (下記)
```

- 置き場は `$XDG_DATA_HOME/haj/trees/<名前>`(既定 `~/.local/share/haj/trees/`)。
  設定でも環境変数でもなく**データ**なので XDG data に置く
- **`--global` はシステム共通に入れる**: `$XDG_DATA_DIRS` の先頭(既定
  `/usr/local/share`)の `haj/trees/<名前>`。イメージに焼くとき
  (Dockerfile の `RUN haj tree install --global ...`)のためにある。
  探索・update・remove は個人とグローバルの両方を見て、同名は個人が勝つ
- **状態ファイルは持たない。git のリポジトリ自体が状態**(URL は remote、版は HEAD)
- git は CLI へ委譲する(op / bao と同じ流儀。git が無ければ実行時にそう言って止まる)
- `@<ref>` でブランチ/タグを固定できる。`@` の後に `/` か `:` があれば URL の一部と
  みなす(`git@host:...` の形と衝突しない)

**ツリーの根**は判定規則ひとつ: `<repo>/.haj` があればそれ、無ければ `<repo>` 自体。
これで「配布専用リポジトリ」(ルート直下に `config` + `commands/` + `docs/` + `lib/`)と
「`.haj/` を持つ普通の haj プロジェクト」のどちらもそのまま入れられる。中の構成は
ユーザー/プロジェクトのスコープと同一(§8.1)で、覚える構造は増えない。
`commands/` は必須ではない — docs だけのツリー(§9.3)も、config だけのツリー
(**エイリアス集の配布**。1行委譲を共有できる)も正当。ただし `commands/` /
`docs/` / `config` の**どれも**無ければ install を拒否する(ゴミを黙って入れない)。

- 名前の決定順: `--name` > ツリーの `config` の `name` > リポジトリ名。
  この名前は**名前空間の名**(§9.7)にも、実行時の `HAJ_TREE`(§3.1)にもなる。
  予約語や既存の語彙と衝突する名前で
  install しようとすると警告する(入れるのは止めない — `--name` で改名すればよい)
- 一覧・which には `[<ツリー名>]` として出る(§2.4)。`docs/` も探索に乗る(§9.3)
- ツリーの `config` で効く鍵はプロジェクト config と同じ発想のホワイトリスト(§2.2)。
  `name` / `alias.*` / `expose`(§9.7)
- **update は黙って入れ替えない**: `git log --oneline <旧>..<新>` を見せる。
  pull は ff-only — ローカルに手を入れたツリーを黙って巻き戻さない
- install は URL を自分で打つ行為なので、それ自体が信頼の表明。update の確認
  プロンプトの要否はツリー信頼ゲート(Issue #1)と歩調を合わせる

**`haj tree configure <名前>` — ツリーの初期値提案。** ツリーは自分に必要な
`tree.*` 設定(宣言や env)の**推奨初期値**を配布物として持てる。ただし
§10.8 の規律は動かない — ツリー自身の設定が勝手に効くことは無く、**保存先は
常にユーザー設定**で、追記の前に本人の確認を挟む(書くことが同意)。

- 規約: ツリーの根に実行ファイル **`config-init`** を置く(`commands/` の外 —
  語彙ではなく、ツリー水準の規約ファイル。docs/ と同じ層)。`haj tree configure`
  はこれを実行し、stdout の提案を表示 → 確認(`y`)→ ユーザー設定へ追記する
- `config-init` の出力書式: `env.KEY = <値>` / `secret.KEY = <参照>` の行と、
  `#` コメント・空行。**インストール名は書かない** — コアが追記時に
  `tree.<インストール名>.` を付ける(多重インストールでもそのまま働く)。
  それ以外の行はエラー(何も追記せず止まる — fail-fast)
- 実行環境: `HAJ_ROOT` / `HAJ_TREE`(インストール名)/ `HAJ_USER_CONFIG` を渡す。
  stdin は `/dev/null`(対話は**コアの確認だけ**に集約する)。stderr は素通し
  (検出の注意書き等)。規約フックではないので §4.5 の制約(2秒・副作用禁止)は
  課さない — 金庫 CLI で自分のユーザー名を引くような**個人化**のためにある
- **既にユーザー設定にある鍵は追記しない**(既存の値を優先し、その旨を表示)。
  提案がすべて設定済みなら何も書かずに終わる
- 確認は stdin から1行読む(`y` のみ追記)。パイプ(`yes |`)も人の明示
- install 完了時、`config-init` があれば1行案内する:
  `初期設定の提案があります: haj tree configure <名前>`
- 追記後の検証はいつもの口: `haj secret check --tree <名前>` /
  `haj config --tree <名前>`(§10.8)

### 9.6 `haj run` — プロジェクトのタスク

プロジェクト局所の作業(install / update / build / test / deploy のような**作業動詞**)を、
コアの語彙と衝突しない名前空間で呼ぶ。

```
haj run                    # タスクの一覧
haj run <名前> [引数...]    # タスクを実行(引数は §11 のとおり無解釈で素通し)
```

**タスクはコマンドではない。** コマンド(§2)は haj の語彙を増やすもの(`mig` のような
ツール)で、どのスコープにも置け、探索と上書きに乗る。タスクは**そのリポジトリの中で
だけ意味を持つ作業**で、探索に乗らない。プロジェクトに `install` コマンドを置くと
`haj install` がコアの動詞(`selfupgrade` / `tree install`)に見えて紛らわしい、という
問題を呼び出しの形で解く: `haj run install` は文脈なしの1行でも「このプロジェクトの
タスク」以外に読めない。

#### 定義できる場所は2つ(エイリアスとコマンドの関係と同型)

| | 1行の宣言(config) | ロジック(実行ファイル) |
|---|---|---|
| 語彙 — 素の `haj <名前>` | `alias.<名前>`(§2.7) | `commands/<名前>`(§2) |
| タスク — `haj run <名前>` | `task.<名前>` | `.haj/tasks/<名前>` |

境界規則も同じ(§2.7): **1行で書けなくなったら実行ファイルに昇格する**。
委譲は宣言、ロジックは実行ファイル。

```
# .haj/config
name = myapp
task.test = exec docker compose exec app vendor/bin/phpunit --testdox
task.test.desc = テストを流す(コンテナ内)
```

`task.<名前>` の意味論はエイリアス(§2.7)と**完全に同じ** — 「`haj` の後に打った語の
並び」へ1回だけ展開され、再帰しない。`.desc` で一行説明、行末 `\` で継続(§8.2)。
つまり config のタスクは「run 名前空間に住むエイリアス」であり、展開器は増えない。

- `task.*` は**プロジェクトの `.haj/config` からだけ**読む(§2.2 のホワイトリスト)。
  ユーザー設定・ツリーの config に書かれた `task.*` は無視される — タスクは
  プロジェクト局所の概念であり、どこでも効かせたいものは語彙(エイリアスまたは
  コマンド)として定義するべきもの
- タスク名の字面の制約は §2.6 と同じ(空・`.` / `-` 始まり・`/` 入りは不可)。
  ただし**予約語は弾かない** — run 名前空間の中に組み込みは居ないので、奪われる
  ものが無い。合成形(`haj help run <名前>`)は位置で決まるため、予約語の名前でも
  紛れない

#### 探索しない・上書きしない・フォールバックしない

- `haj run <名前>` が見るのは現在のプロジェクトの `task.<名前>` と `.haj/tasks/<名前>`
  **だけ**。親へは遡らず(`root = false` でも)、ユーザー/ツリー/共通/PATH も見ない
- 同名が両方に定義されていれば **`task.<名前>`(宣言)が勝つ**
  (予約語 > エイリアス > ツリー名前空間 > 探索、と同じ「宣言が手前」の並び)
- 未定義の名前は exit 127。同名のコマンドが探索に居る場合は、実行せずに案内する:
  `install はタスクではありません。コマンドとして定義されています: haj install`
- プロジェクトの外(遡って `.haj` が見つからない)では、その旨を出して exit 1

この制約が保証そのもの: **`haj run x` という1行は、どの環境で読んでも「そのリポジトリの
`task.x` / `tasks/x` が走る」以外に解釈できない。** 逆に素の `haj <名前>` の意味論は
何も変わらない — プロジェクトの `commands/` も従来どおり素の形で呼べ、手前勝ち(§2.3)も
そのまま残る。

#### タスクファイルの契約はコマンドと同じ

`.haj/tasks/<名前>` の扱いは `.haj/commands/<名前>` と対称:

- 実行可能ファイルの判定は §2.5、実行は §3(exec(2) で置き換え、終了コードは素通し)
- 規約フック(§4)がそのまま乗る: `--haj-describe` は `haj run` の一覧に、`--haj-help`
  は `haj help run <名前>` に、`--haj-env` は `haj env run <名前>` に出る。説明文
  キャッシュ(§7)も同様に効く
- `haj env run`(名前なし)は、応答する**全タスク**の `--haj-env` を
  `# ==== <名前> ====` の節で連結して出す。出力はそのまま `--env-file` に渡せる
  形式のまま(§4.4)。1行宣言(task.*)は委譲であって環境変数を持たないので出ない
- 環境変数(§3.1)も同じ: `HAJ_ROOT` はそのプロジェクトの `.haj`、`HAJ_NAME` は
  タスクの名前(`run` ではない)、`HAJ_PROJECT` / `HAJ_PROJECT_DIR` は通常どおり
- シークレット(§10)のフラグはグローバルフラグなので今までどおり名前の前に書く:
  `haj --secret DB_PASS=vault://secret/data/db/password run migrate`

`haj which run <名前>` は、効いている定義(`task.*` の展開、またはタスクファイルの
パス)を表示する(`--all` で両方)。

#### 使い分けの指針

- **リポジトリの作業動詞**(install / update / build / test / deploy / setup)→ タスク
- **haj の語彙を増やすツール**(`mig` のような、名前がそのまま意味になるもの)→ コマンド
- 迷ったら: その名前を素の `haj <名前>` で読んだとき、コアの動詞や他スコープの同名と
  紛れるか?紛れるならタスク

### 9.7 ツリー名前空間 — `haj <ツリー名> <名前>`(2段構えの語彙)

タスク(§9.6)は「プロジェクト局所の作業動詞」を素の語彙から分けた。同じ問題は
**ツリー**でも起きる — 横断的な道具に install / update のような汎用動詞を付けて配ると、
素の語彙が紛らわしくなる。ツリーはプロジェクトではないので run には入れられない。
そこでツリー名を名前空間にする。

```
haj <ツリー名>                    # そのツリーのコマンド一覧(run と同じ振る舞い)
haj <ツリー名> <名前> [引数...]    # そのツリーの commands/<名前> を実行
```

- **明示形は全ツリーで常に使える**(下の expose に関係なく)。run が「このプロジェクトの」
  を明示するのと同型で、こちらは「このツリーの」を明示する
- 実行の意味論は素の形と同じ: `HAJ_ROOT` はそのツリー、`HAJ_NAME` は `<名前>`
  (ツリー名ではない)。規約フック(§4)・キャッシュ(§7)・シークレット(§10)も同様
- 合成形も同じ: `haj help <ツリー名> <名前>` / `haj env <ツリー名> <名前>` /
  `haj which <ツリー名> <名前>`。補完は §6
- `haj env <ツリー名>`(名前なし)は、応答する**全コマンド**の `--haj-env` を
  `# ==== <名前> ====` の節で連結して出す(そのツリーが読む環境変数の全景。
  出力は `--env-file` に渡せる形式のまま — §4.4)
- 未知の名前は exit 127。名前の字面の制約は §2.6 と同じだが、run と同様に
  **予約語は弾かない**(名前空間の中に組み込みは居ない)。`haj <ツリー名> tree` の
  ような自然な名前を予約語が塞ぐ方が害になる。なお素の探索(§2)では従来どおり
  予約語の名前は無視される — 予約語が守っているのは素の名前だけ

#### `expose` — 素の露出はツリーごとに選ぶ

ツリーの `config` に書ける(ホワイトリストの一員。§9.5):

```
# ツリーの config
name = ext
expose = namespace
```

| 値 | 意味 |
|---|---|
| `flat`(既定) | 従来どおり。コマンドは素の探索(§2 の順 3)にも乗る |
| `namespace` | 素の探索から**外れる**。名前空間(`haj <ツリー名> <名前>`)でだけ呼べる |

install / update のような**汎用動詞**を配るツリーは `namespace` を宣言する。素の語彙を
汚さず、`haj ext install` という1行はコアの動詞とも他のツリーとも紛れない(§9.6 と同じ
発想のツリー版)。名前がそのまま意味になるコマンド群(`cert` の類)は既定のままでよい。

- `expose` が効くのは **commands の素の露出だけ**。docs(§9.3)とエイリアス(§2.7)は
  従来どおり探索に乗る
- 既定が flat なので、既存のツリーの挙動は変わらない
- `haj help` では、namespace のツリーはコマンドを並べず
  「`<ツリー名>` … (haj <ツリー名> で一覧)」の1行に畳む。`haj tree list` には
  `(namespace)` と出る

#### 名前解決の順(v3)

名前の位置は次の順で解決される:

**予約語(組み込み) > エイリアス > ツリー名前空間 > 探索**

- ツリー名は install するとき自分で選ぶ(`--name`。git remote と同じで、名前空間の名は
  使用者の管理下)。予約語や既存の語彙と衝突する名前は install 時に警告される(§9.5)
- ツリー名が探索より手前なのは、名前空間が **cwd に依存せずどこでも同じに動く**ため。
  探索を手前にすると、あるプロジェクトに同名コマンドが生えた瞬間、そのリポジトリの中で
  だけ `haj <ツリー名> ...` が別物になる — 名前空間の意味が場所で揺れるのは §2.4 に反する
- 逆に、ツリー名と同名のコマンドは名前空間に隠れる。`haj which --all <名前>` が両方を
  見せ、困るなら remove して `--name` で入れ直せばよい

これで名前の位置は3種に整理される:

| 呼び出し | 意味 | 中身 |
|---|---|---|
| `haj <名前>` | 語彙 | 自分のスコープに置いたもの + flat ツリー + 共通 + PATH |
| `haj run <名前>` | このプロジェクトの作業(§9.6) | `task.*` / `.haj/tasks/` |
| `haj <ツリー名> <名前>` | そのツリーの道具(§9.7) | `<ツリー>/commands/` |

## 10. シークレット参照(展開)

サブコマンドに渡す値には、シークレットの実体ではなく**参照**を書ける。コアは exec の
直前に参照を解決し、展開済みの値を子プロセスに渡す。リポジトリにも環境にも平文を置かず、
haj がプロキシになる。

**参照の書式は発明しない。** 1Password は `op inject` の書式、Vault は vault-agent の
template で使う展開式を、それぞれ**そのまま**受け付ける。既存のテンプレートや
ドキュメントからコピペで移せることを設計要件とする。

### 10.1 シークレットは人が明示的に渡すものだけ

haj は**環境を勝手に走査しない**。渡すものと相手を、人がその実行時に明示する。

```sh
haj --secret DB_PASS=vault://secret/data/db/password mig up
```

環境変数を全走査して参照を展開する仕組み(0.13.0 までの `HAJ_SECRETS=1`)は**廃止した**。
「値が参照かどうか」しか見ておらず「**誰に**渡すか」を見ていないため、
DB パスワードを必要としないコマンドにも実体が渡り、信頼していないツリーのコマンドにも
届いてしまう。参照をたまたま文中に含むだけの変数(CI が置く説明文など)で誤爆もする。

### 10.2 渡し方(グローバルフラグ)

フラグは**サブコマンド名の前**にだけ書ける(§3.2)。フラグを打ったこと自体が同意。

| フラグ | 意味 |
|---|---|
| `--secret <名前>=<値>` | 展開して環境変数 `<名前>` で渡す。参照でなければ平文のまま |
| `--env-file <ファイル>` | `key = value` を読み、各値を展開して渡す |
| `--secret-file <名前\|パス>=<参照\|テンプレート>` | 値をファイルに書く(§10.4) |

**`--secret-file` は「ファイルで渡せ」と要求するツール向け**(ssh の鍵、kubeconfig、
GCP のサービスアカウント JSON、`.npmrc` など)。

| 右辺 | 中身 |
|---|---|
| 参照(`vault://` / `op://` / `{{...}}` / `env://` / `file://`) | その**値**がファイルの中身になる |
| それ以外 | **テンプレートファイル**のパスとみなして描画する |

| 左辺 | 書き先 |
|---|---|
| **名前**(環境変数として妥当なもの。`KEY` `KUBECONFIG` など) | **一時ファイル**に書き、そのパスを**環境変数 `<名前>`** に入れる |
| **名前/相対パス**(先頭が大文字の名前。`GLAB_CONFIG_DIR/config.yml`) | 一時**ディレクトリ**の中に書き、ディレクトリのパスを**環境変数 `<名前>`** に入れる |
| **パス**(それ以外。`config.ini` `~/.npmrc` `./out` など) | そのパスに書く(親ディレクトリは作らない) |

```sh
haj --secret-file KEY=vault://secret/data/ssh/id_rsa sh 'ssh -i "$KEY" host'
haj --secret-file KUBECONFIG=vault://secret/data/k8s/config exec kubectl get pods
haj --secret-file ~/.npmrc=vault://secret/data/npm/rc exec npm publish
haj --secret-file config.ini=config.ini.tpl app run     # テンプレート描画
haj --secret-file GLAB_CONFIG_DIR/config.yml=config.yml.tpl exec glab mr list
```

左辺が名前のとき、環境変数名は**ツールが要求するものをそのまま書けばよい**
(`KUBECONFIG` / `GOOGLE_APPLICATION_CREDENTIALS` など)。

左辺が `<大文字の名前>/<相対パス>`(`GLAB_CONFIG_DIR/config.yml`)なら、一時
**ディレクトリ**の中の相対パスに書き、ディレクトリのパスを環境変数に入れる —
「設定ディレクトリを環境変数で指せ」と要求するツール(glab など)にそのまま嵌まる。
複数のファイルを同じ名前で指定すれば同じディレクトリに並ぶ。先頭セグメントに
小文字が混ざるものは従来どおり相対パスとして扱う(`out/config.ini` を奪わない)。

- 一時ファイルは `$XDG_RUNTIME_DIR`(無ければ `$TMPDIR`)の mode 0700 ディレクトリに
  mode 0600 で作る。**cwd には決して書かない** — リポジトリに置かれて commit される
  事故を防ぐため
- **haj はこのファイルを消さない。** コアは exec(2) で自分を置き換えるため、実行後の
  後始末は構造的に不可能。消すのは使う側の責任
- 全て解決できてから書く。失敗時は書かない(半端なファイルを残さない)

- 同じ変数への指定が重なったら**書いた順で後勝ち**
- 規約フック(`--haj-describe` 等)には**展開しない**。TAB のたびに金庫へ問い合わせが
  飛ぶのは論外(タッチ認証で固まる)。展開は「本体を実行する」ときだけ
- **常用するツリーの定常は設定に書ける**(§10.8)。平文の設定は
  `tree.<インストール名>.env` で注入され、秘密は `tree.<インストール名>.secret` に
  **宣言**してコマンドが `haj secret get` で引く(注入されない)。フラグは
  「その実行だけの明示」で、常に勝つ
- プロジェクト層(`.haj/env`)は未実装。入るときはツリー信頼のゲートとセット

### 10.3 参照の書式

| 参照 | 意味 |
|---|---|
| `op://<金庫>/<アイテム>/[<セクション>/]<フィールド>` | 1Password。**`op inject` に丸ごと委譲**し、書式も意味論も inject に従う |
| `{{ with secret "<パス>" }}{{ .Data.data.<フィールド> }}{{ end }}` | Vault。vault-agent template の正準形。KV v2 の `/data/` 入りパスをそのまま書く |
| `vault://<パス>/<フィールド>` | 上の短縮形。**最後のセグメントがフィールド**、残りがパス(パスの規約は template 形と同じ) |
| `env://VAR` | 環境変数 VAR の値。再帰はしない(1段だけ) |
| `file://<パス>` | ファイルの中身。docker secrets / systemd credentials との接続に |
| `store://<論理パス>` | **そのコマンドが属するツリー専用**の名前空間(§10.7)。最後のセグメントがフィールド。ツリー文脈が無ければエラー |

**環境変数の走査は、値全体が参照のときだけ**展開する(`op://` / `vault://` 等で始まる、
または `{{` で始まり `}}` で終わる)。文字列中への埋め込みは解釈しない。接続文字列の
組み立てはサブコマンド側の責務。

埋め込みを許すと「参照をたまたま文中に含む」だけの変数まで解決しに行ってしまう。
実例: GitLab の MR パイプラインは `CI_MERGE_REQUEST_DESCRIPTION` に MR の説明文を
入れる。説明文に `op://` の例が書いてあるだけで、op の解決が走って全体が止まる。

**コマンドライン引数では op の埋め込みも展開する**(inject の意味論のまま)。
argv は人がその実行のために明示的に書いたものなので、誤爆の心配がない。

```
DB_PASSWORD=vault://secret/data/app/db_password                               → 展開される
OTP={{ with secret "secret/data/app" }}{{ .Data.data.db_password }}{{ end }}  → 展開される(上と等価)
TOKEN=op://Infra/ci/token                                             → op inject が展開
DB_URL=postgres://u:vault://a/b/c@host/db                             → 展開されない(ただの文字列)
NOTE="説明: op://a/b/c を参照…"                                        → 展開されない(値全体が参照ではない)

haj deploy --token "Bearer op://Infra/ci/token"                       → 引数なので埋め込みも展開される
```

vault の template 形は **`with secret` ブロックだけ**を解釈する。ブロックの中には
地の文と複数の `{{ .Data.data.<フィールド> }}` を書け、Go template の空白制御
(`{{-` / `-}}`)も効く — vault-agent で使っているテンプレートがそのまま動く。
`printf` / `range` 等それ以外の式が来たら解釈できないと言って中止する
(テンプレートエンジンは抱え込まない)。

### 10.4 解決

依存クレートは増やさない。op / vault は CLI を子プロセスとして呼び、env / file は
stdlib だけで解決する。

| 参照 | 解決方法 |
|---|---|
| op | `op inject` に値を stdin で渡し、stdout を採る |
| vault | `vault kv get -field=<フィールド> [-mount=<マウント>] <パス>`。パスの2セグメント目が `data` なら KV v2 の API パスとみなし、`-mount=<先頭>` と相対パスに読み替える |

CLI は差し替えられる(§8.3): `HAJ_OP_CMD` / 設定 `secrets.op_cmd`(既定 `op`)、
`HAJ_VAULT_CMD` / 設定 `secrets.vault_cmd`(既定 `bao`)。

**vault サーバ**: 環境に `VAULT_ADDR` / `BAO_ADDR` があればそれを尊重する。無ければ
設定 `secrets.vault_addr`(既定 `https://vault.example.com`)を両方の名前で CLI に渡す
(bao は `BAO_ADDR` を先に見る)。

**vault の自動ログイン**: 未ログインのとき(`token lookup` が非 0)に限り、解決の前に
`login` を**端末を継いで**実行する。引数は設定 `secrets.vault_login`(環境変数
`HAJ_VAULT_LOGIN`)から取り、既定は `-method=oidc`。

```
# ~/.config/haj/config — 既定から変えたいときだけ書く
secrets.vault_login = -method=oidc -path=<パス>
```

- `secrets.vault_login = off` で無効化する。そのときは解決が vault 自身のエラーで
  fail-fast する(従来どおり)
- ログイン状態の確認は 1 プロセスにつき 1 回だけ。ログイン済みなら何も起きない
- 引数は空白区切りで分割する(引用符やエスケープは解釈しない)
- **認証しない CI で vault 参照を使うなら `HAJ_VAULT_LOGIN=off` を置くこと。**
  OIDC ログインはブラウザと人を待つ。`VAULT_TOKEN` 等で認証済みの CI では
  `token lookup` が通るので login は走らない

- **タイムアウトは設けない。** op のタッチ認証など、人を待つ場面が正当にある。
  規約フックの 2 秒とは別物。
- 展開結果の末尾の改行 1 つは落とす(CLI や credential ファイルが付けるもの)。
- **キャッシュしない。** 毎回聞く。セッション管理は各 CLI の仕事。

### 10.5 失敗したら止まる

解決に失敗した(CLI が無い・非 0 で終わった・参照の書式が壊れている・変数やファイルが
無い)ら、**本体を実行せずに exit 1**。未解決の参照文字列が DB パスワードとして
そのまま使われ、分かりにくく壊れる事故を防ぐ(fail-fast)。

### 10.6 `haj secret check`(dry-run)

何が渡り、何が宣言されているのかを、**解決せずに**確かめる。金庫に問い合わせない
ので、OIDC ログインもタッチ認証も起きない(参照そのものは秘密ではない)。

**受け渡しフラグの事前確認**(何が渡るのか):

```console
$ haj --secret DB_PASS=vault://secret/data/db/password --env-file ./mig.env secret check
 実行時に渡るもの (値は解決していません):
   --secret    DB_PASS               → vault://secret/data/db/password
   --env-file  DB_HOST                 db.example.internal
   --env-file  DB_USER               → vault://secret/data/db/user

 (→ が付いたものが展開されます。他は平文としてそのまま渡ります)
```

**宣言の検証**(ツリー文脈 — 環境の `HAJ_TREE` — があるとき): 宣言(§10.8)の
各参照の書式を検証し、`store://` には展開先の物理写像を添える。参照でない値
(平文の宣言)はここでエラーとして見える。

```console
$ haj secret check --tree hajime
 宣言 (tree.hajime.secret.*):
   CLIENT_SECRET  → vault://secret/data/google/oauth/client_secret
   MM_TOKEN       → store://token  (→ vault://secret/data/users/alice/trees/hajime/token)
```

対象の明示は `--tree <インストール名>`(人手用 — §10.9)か、環境の `HAJ_TREE`
(ツリーのコマンドの中ではコアが注入している)。`--tree` が手前。

ストア参照の写像は手元の設定だけで決まる — どちらの検証も金庫には触らない。
ツリー文脈が無ければ写像の形(`<prefix>/trees/<HAJ_TREE>/...`)だけを示す。

**`haj secrets`(旧名)は移行スタブ。** 呼ばれたら `haj secret check` への改名を
案内して exit 1。予約語には残す(§2.6 — 名前を探索に明け渡すと、旧名で書かれた
スクリプトが野良コマンドに落ちる)。スタブの削除は将来の緩和(契約は据え置き)。

### 10.7 ストア — ツリー専用の名前空間(`store://`)

ツリーのコマンドが**実行時に得る**秘密(OAuth の refresh token など)には、
**書き込み先**が要る。ローカルに平文で永続するのは論外。かといってコマンドに
`vault://` の物理パスを書けば、物理(サーバ・マウント・パス)がコードと README に
散らばるうえ、多重インストール(§3.1 の `HAJ_TREE`)ではインスタンス同士が
**同じパスを奪い合う**。

**ストア**は、エンジン(金庫)の中に haj が確保する、**ツリー・インスタンス専用の
名前空間**である。参照は自分の論理パスだけを書き、置き場所は haj が決める:

```
store://<論理パス>   →(展開)→   <prefix>/trees/<HAJ_TREE>/<論理パス>
```

例(すべて既定のまま、ツリー `hajime` のコマンドから):
`store://token` → `vault://secret/data/users/alice/trees/hajime/token`。
書式の規則は `vault://` と同じで、**最後のセグメントがフィールド**
(`store://google-oauth/token` はオブジェクト `google-oauth` のフィールド `token`。
同じオブジェクトに複数フィールドを並べられる)。

- **ツリーは自分の名前を書かない。** `store://<ツリー名>/...` のような明示形は
  **無い** — 同じコードがどの名前でインストールされても、自分のインスタンスの
  名前空間だけを指す。**ツリーをまたぐ参照は構文レベルで存在しない**。またぎたい
  秘密は物理参照(`vault://`)で渡す(将来拡張も参照)
- **文脈が要る。** 展開には「どのインスタンスのための参照か」= `HAJ_TREE` が要る。
  受け渡し(§10.2)では**渡す相手のコマンドが属するツリー**、宣言の解決と
  `haj store`(§10.9 / §10.10)は**自分の環境の `HAJ_TREE`**。文脈が無い
  (プロジェクトや個人のコマンドに渡す・`exec` / `run` に渡す・シェルから直に叩く)
  場合は**エラー** — 点検・横断のやり方(§10.10)を案内する
- **ストアの設定は表である**: `store.<名前>.{engine, prefix}`。v1 で存在する行は
  **予約された `tree`**(`store://` が指すストア)だけで、設定キーは
  `store.tree.engine` / `store.tree.prefix`。将来の名前付きストア(下記の将来拡張)は
  この表に行を足すだけで、キーの形は変わらない。`tree` という名前は予約 —
  将来ストアを定義できるようになっても、この行の意味(自ツリーの名前空間)は
  ユーザー定義で置き換えられない
- **engine は v1 では型名 `vault` を直接書く**(唯一のエンジン)。将来エンジンの
  定義が増えたら、この値は**エンジン名の参照**になりうる(型名はエンジン定義側へ
  移る)。接続・認証・セッションは物理参照(§10.4)と**同じ機構**
  (`secrets.vault_*` と自動ログイン)をそのまま使う。ストアが足すのは名前空間の
  写像だけで、金庫との話し方は増えない
- `store.tree.prefix` の書式は物理参照のパスと同じ(KV v2 の `/data/` 入り)。既定
  `secret/data/users/<ユーザー名>` の `<ユーザー名>` は実行ユーザーから採る
- **解決は一段だけ。** 解決結果は常にリテラル — 値がまた参照の形をしていても解釈
  しない(連鎖参照の迷路を作らせない。`env://` の1段規則と同じ)
- **値はキャッシュしない**(§10.4)。オフライン耐性が要るなら、「金庫から読めたら
  自分の状態に写す」二段構えを利用側が組む — それは秘密ごとの設計判断で、コアが
  一律に肩代わりするものではない

将来のエンジン(`file` / `age` / `pass`、1Password の `op`)はエンジンの型の
追加で足す(追加機能につき契約は据え置き)。エンジンが定義するのは
get / put / login / status の4動詞への写像だけで、参照の書式・fail-fast・
キャッシュしない、はストアの層で共通。プラグイン機構は作らない(§11)。

**将来拡張(検討中): またぎ用途の名前付きストア。** ツリー横断・共有の秘密に論理名を
与える形(表に行を足す — `store.<名前>.{engine, prefix}` — と `<名前>://` の参照)は、
実需を見て別途入れる。v1 で
またぎに物理参照を使っている箇所が増えて散らばりが痛くなったら、が入れどき。
そのときに「定義済みの名前だけをスキームとして扱う」等の字面規則をこの節に足す。

### 10.8 ツリーごとの設定(`tree.<インストール名>.env` / `.secret`)

常用するツリーに毎回フラグを打つのは億劫で、いずれ打ち間違える。ユーザー設定に
そのインストール名のための行を書いておく。**env と secret は渡り方が違う** —
平文の設定は環境変数として注入されるが、**秘密は環境に勝手に載らない**。
コマンドが要る瞬間に自分で引く(pull — §10.9)。

```
# ~/.config/haj/config
tree.hajime.env.GOOGLE_ACCOUNT   = hajime@example.com
tree.hajime.env.TOKEN_OUTPUT     = token               # 論理パスもただのデータ
tree.hajime.secret.CLIENT_SECRET = vault://secret/data/google/oauth/client_secret
tree.avap.env.GOOGLE_ACCOUNT     = m@avap.example      # 別インスタンスには別の値
```

| 鍵 | 値 | 渡り方 |
|---|---|---|
| `.env.KEY = <値>` | 平文 | exec 時に環境変数 `KEY` として**注入**。一切展開しない(`store://` と書いてあってもただの文字列) |
| `.secret.KEY = <参照>` | 参照(§10.3) | **宣言**(capability)。exec 時には**何もしない**。コマンドが `haj secret get KEY` で引いたときだけ解決される(§10.9) |

- **宣言は「このツリーが扱える秘密の目録」である。** コアは宣言に無い参照を
  解決しない(`haj secret get` は表を引くだけ — §10.9)し、宣言があっても env には
  載せない。秘密が環境に現れるのは、人がフラグ(§10.2)で明示したときだけ —
  「渡すものと相手をその実行時に明示する」(§10.1)が、定常化しても崩れない。
  目録は `haj secret list` で、検証は `haj secret check`(§10.6)で見える
- `.env` が展開しないのは仕様であって手抜きではない — 参照や論理パスを
  **データとして**渡せる(上の `TOKEN_OUTPUT=token` を受けたコマンドは、
  `haj store put "$TOKEN_OUTPUT"` のように**書き先の指定**として使う。§10.10)
- `.secret` に参照でない値が書いてあったら、`haj secret get` / `check` はエラー —
  秘密の平文を設定ファイルに書かせない(平文の設定は `.env` に書くもの)
- **`.env` の優先順位**: グローバルフラグ(§10.2)> シェル環境 > tree設定 >
  コマンド自身の既定値(`${VAR:-...}`)。tree設定は**その変数が未設定のときだけ**
  注入する — その場の明示が常に勝つ。**`.secret` に優先順位は無い**(注入しない
  ので競合しない)。その実行だけ別の値を使いたいときのために、コマンド側の定石は
  環境変数を先に見ること:

```sh
token="${MM_TOKEN:-$(haj secret get MM_TOKEN)}"   # haj --secret MM_TOKEN=... が勝てる
```

- **権威はユーザー設定だけ。** ツリー自身の `config`・プロジェクトの `.haj/config` に
  書かれた `tree.*` は無視される — clone した木が自分の宣言を書けると、任意の
  物理パスを目録に足して `haj secret get` で吸い出せてしまうし、`.env` は書き先
  (論理パス)のすり替えになる(§2.2 のホワイトリストと同じ理屈)。逆に、
  `~/.config/haj/config` は本人にしか書けないファイルなので、**書いたこと自体が
  同意** — フラグを打つのと同じ層(§8.4 の `selfupgrade.token` と同じ)。環境の
  走査(§10.1)はしていない — 注入も宣言も、本人が名指しで書いた鍵だけ
- 規約フック(§4)に注入されるのも `.env` だけ(本体と同じ)。`.secret` には
  そもそも「注入」という経路が無い — フックにも本体にも、env には載らない
- 入口: 手で書く(`haj config --init` の雛形に例が出る)ほか、ツリーが初期値を
  **提案**できる — `haj tree configure <名前>`(§9.5)。どの入口でも
  **保存先は常にユーザー設定**で、追記の前に本人の確認を挟む

**`haj env <名前>` の注記は `.env` の分**(§4.4 の中継に、コアが知っている出所を足す):

```console
$ haj env hajime gmail
GOOGLE_ACCOUNT=hajime@example.com   # tree設定 (env)
TOKEN_OUTPUT=token                  # tree設定 (env)
LONG_TX_SEC=60
```

シェル環境が勝っている鍵には `# シェル環境` が付く。注記は行末コメントなので、
出力は従来どおり `--env-file` にそのまま渡せる(§4.4)。**秘密の目録は env とは
別物**になったので、`haj env` ではなく `haj secret list`(§10.9)が受け持つ。

**そのインスタンスの全景**は `haj config --tree <インストール名>` で一覧できる
(人手の点検用): `.env` は実効値と出所(シェル環境 / 設定ファイル)、`.secret` は
**参照のまま**、store の名前空間(§10.7 の写像先)、続けて**実効 env** —
そのツリーの全コマンドの `--haj-env` の節連結(上の中継と同じもの。コマンド自身の
既定値まで含み、出所注記も付く)。既定値の権威はコマンド本人にあるので(§4.4)、
コアは設定ファイルを写すのではなく規約フックで本人に聞く — 一覧は実態と一致する
(§2.4)。`haj secret list` / `check` と同じく、**金庫には触らない**(実行するのは
`--haj-env` フックだけ。§4.5 の副作用禁止がここでも効く)。`--env-file` に渡せる
素の形式が要るときは従来どおり `haj env <ツリー名>`(§9.7)。

### 10.9 `haj secret` — 宣言を引く(読みだけ)

サブコマンドが金庫を**直接**読むと、接続と認証の知識(アドレス・認証方式・
ログインの手順)がツリーごとに複製される。`haj secret` はその口を一つにする —
ただし解決するのは**宣言表(§10.8)にある参照だけ**である。

```
haj secret get <KEY>                       # 宣言 tree.<HAJ_TREE>.secret.<KEY> を解決して stdout へ
haj secret list  [--tree <インストール名>]  # 宣言の一覧(KEY=<参照>。値は解決しない)
haj secret check [--tree <インストール名>]  # 宣言と受け渡しの検証(§10.6。金庫に触らない)
```

```sh
# ツリーのコマンドの中(HAJ_TREE はコアが注入している — §3.1)
client_secret=$(haj secret get CLIENT_SECRET)
```

- **KEY で引く。参照は受けない。** 何を読めるかは宣言表が決める(capability)。
  ツリーのコードには物理パスも `store://` も書かれず、宣言表を**迂回する口が無い**
- 文脈は**自分の環境の `HAJ_TREE`**(§10.10 と同じ)。ツリーの外ではエラー
- **`list` / `check` だけは人手用の `--tree <インストール名>` で対象を明示できる**
  (人手の点検は日常の操作 — `HAJ_TREE=` の前置きを覚えさせない)。明示は環境より
  手前: `--tree` > 環境の `HAJ_TREE`(§10.8 の「その場の明示が常に勝つ」と同じ並び)。
  **`get` には無い** — 値に触る操作の対象切り替えを argv で気軽にさせない
  (取り違えたツリーの秘密を読む事故)。`list` / `check` は金庫に触らない
  **読み取りメタ情報**(参照は秘密ではない — §10.6)だから、この口が許される。
  `get` の人手は従来どおり `HAJ_TREE=<インストール名>` の明示で(§10.10)
- 宣言に無い KEY はエラー(宣言済みの KEY を列挙して案内する。fail-fast — §10.5)
- `get` の出力は値そのもの+改行1つ(`$(...)` が改行を落とす)。値の末尾の改行の
  扱いは §10.4 と同じ
- **読みだけ。** 宣言の参照先は他所の所有物でありうる(共有の金庫など)。haj から
  書く口は無い — 書きたい秘密は**自分の store**(§10.10)に置く。これが所有の規律:
  **secret = 読む(他所の物も含む)、store = 読み書き(自分の物だけ)**
- 補完(§9.0): 動詞と、`get` には**宣言済みの KEY**(目録は手元の設定だけで
  列挙できる — 金庫には触らない)
- `secret` は予約語(§2.6)。奪われると「get は宣言された参照しか解決しない」という
  この節の保証が破れる(偽の値の差し込み)

### 10.10 `haj store` — 自ツリーのデータの読み書き

コマンドが実行時に得た秘密(トークン等)を置き直す先。**store は常に自分の
名前空間(§10.7)を操作する**ので、引数は裸の論理パスでよい。

```
haj store get <論理パス>            # 値を stdout へ(末尾に改行1つ)
haj store put [--force] <論理パス>  # stdin から値を読んで書く
haj store login                     # エンジンにログインする(vault なら §10.4 の login)
haj store status                    # ログイン状態と実効設定(engine / prefix / 接続先)
```

```sh
# ツリーのコマンドの中。自分がどのインスタンスかは知らないまま
printf '%s' "$refresh_token" | haj store put token
token=$(haj store get token)
```

- 論理パスは `store://` 前置きでもよい(読み飛ばすだけ)— 設定から**データとして**
  受け取った参照(`TOKEN_OUTPUT=store://token` — §10.8)をそのまま渡せる
- **物理参照(`vault://`)は受けない**(下記)。`haj secret`(§10.9)と併せて、
  ツリーのコードが名前空間・宣言表を迂回して物理を固定で書く道を、コアは舗装しない
- 文脈は**自分の環境の `HAJ_TREE`**。ツリーの外ではエラー(人手の操作は下記)
- **`put` は stdin からだけ受ける。** argv に実体を書く口は**作らない**(`ps` /
  `/proc/*/cmdline` / シェル履歴に残る — §10.11)。stdin が TTY ならエコーを切って
  入力を促す。末尾の改行1つは落とす(`echo x | haj store put token` が `x` を書く)
- **`put` はフィールド単位の patch**(`kv patch`。オブジェクト自体が無ければ
  `kv put`)— 同じオブジェクトの他のフィールドを壊さない。**既にフィールドが在れば
  `--force` 無しでは拒否**する。上書きは明示 — 確認プロンプトは設けない(スクリプト
  から使う plumbing に対話を混ぜない。対話的な安全は「まず拒否される」ことで足りる)
- 補完(§9.0): 動詞を返す。**論理パスは補完しない** — TAB のたびに金庫へ
  問い合わせない(§10.2 と同じ理由)
- `store` は予約語(§2.6)。奪われると `put` の stdin を横取りされる — `exec` / `sh`
  と同じ層

**点検・横断・移行は、ツリー文脈の外で・人の明示で行う**(ツリーのコードに
書かせない、が上の規律。人手の道は3つで足りる):

```sh
# 自分の store への人手の操作: HAJ_TREE を明示する
#(シェルで環境変数を打つのは人の明示 — フラグと同じ層)
HAJ_TREE=hajime haj store get token

# 任意の参照の読み(点検): 受け渡しフラグはどの参照でも解決できる(§10.2)
haj --secret V=vault://secret/data/x/token sh 'printf "%s\n" "$V"'

# 任意の物理パスへの書き(移行): エンジンの CLI で
bao kv put ...
```

写像の点検は `haj secret check`(§10.6)と、エラー時の物理写像の提示で足りる。

**エラーは3状態を区別する**(どれも fail-fast。§10.5):

| 状態 | 出すもの |
|---|---|
| 文脈なし(`HAJ_TREE` が無い) | `haj store はツリーのコマンドの中でだけ使えます` + 人手の操作は `HAJ_TREE=<インストール名>` を明示する案内 |
| 未ログイン | 自動ログイン(§10.4)が無効なら、エンジンの login を案内して止まる |
| パス不在・その他 | エンジン CLI の stderr をそのまま流し、haj が**展開後の物理写像**を添える: `token → bao kv get -mount=secret -field=token users/alice/trees/hajime (exit 2)`。名前空間で隠した物理は、**失敗の瞬間には見えなければならない**(§2.4) |

### 10.11 注意(仕様ではなく性質)

- argv に展開したシークレットは `ps` / `/proc/*/cmdline` から見える。**シークレットは
  環境変数で渡すこと**。argv 展開は「参照を引数で受けたい」場面のためにあるが、
  実体を argv に晒す危険は使う側が負う。
- 展開後の値は子プロセスの `/proc/<pid>/environ` に見える(環境変数渡しの本質的な限界)。

## 11. 意図的にやらないこと

- **タスクのロジックを宣言型ファイル(`haj.toml` の `run = "..."`)で書けるようにしない。**
  分岐や冪等性判定を含む現実のタスクは宣言型に収まらず、「シェルスクリプトを
  TOML の文字列に埋め込んだもの」に堕する。タスクはファイルとして書く。
  `alias.*` / `task.*`(§2.7 / §9.6)は**1行の委譲**であってこれの例外ではない —
  1行で書けなくなった時点で実行ファイルに昇格する、という境界規則がこの原則を守る。
- **コアがサブコマンドの引数を解釈しない。** 引数はそのまま渡す。コアが知るのは
  「名前」だけ。
- **サブコマンドの実装言語を制約しない。** 契約は exec と引数と stdout だけで定義され、
  言語に依存しない。これによりコアを将来別の言語で書き直しても、サブコマンドは
  1 行も直さずに動く。
