# haj 仕様

このドキュメントは **haj コアと、haj サブコマンドの間の契約** を定める。

サブコマンドの実装言語は問わない。この契約さえ満たせば、シェルでも Rust でも PHP でも
Python でも `haj` から等しく扱われる。逆に、コアはこの契約以外の方法でサブコマンドの
中身を知ろうとしない。

契約バージョン: **1**(`haj --version` はコアの版であって契約の版ではない)

---

## 1. 用語

| 語 | 意味 |
|---|---|
| コア | `haj` 実行ファイル本体。探索と exec だけを行う |
| サブコマンド | `haj <名前>` で呼ばれる実行可能ファイル |
| ツリー | `commands/` を含むディレクトリ。サブコマンドには `HAJ_ROOT` として渡される |

---

## 2. 探索

コアはディスパッチ表を持たない。`haj <名前>` は、以下の順で **実行可能ファイル** を探し、
**最初に見つかったもの**を実行する。

| 順 | 探索先 | 用途 |
|---|---|---|
| 1 | カレントから上へ辿った各階層の `<dir>/.haj/commands/<名前>` | プロジェクト固有 |
| 2 | `~/.config/haj/commands/<名前>` | 個人用(XDG。§8.1) |
| 3 | `$HAJ_COMMAND_PATH` の各ディレクトリ(`:` 区切り) | 全社/イメージ共通 |
| 4 | `$PATH` 上の `haj-<名前>` | git 方式の逃げ道 |

`$HAJ_COMMAND_PATH` の既定値は `/usr/local/lib/haj/commands`(設定ファイルの
`command_path` でも指定できる。§8)。

### 2.1 プロジェクト境界(`.haj` は壁である)

**1 の遡上は `/` までは行かない。`.haj` を持つディレクトリで止まる。**

素朴に `/` まで遡って見つかった `.haj/commands` を全部積むと、二つの困りごとが起きる。

1. **上流の野良 `.haj` が黙って効く。** 誰かが `~/repos/.haj/commands/setup` を置くと、
   その配下の**全リポジトリ**で `haj setup` が生えてしまう。置いた本人以外は気づけない。
2. **どのプロジェクトの `setup` が走ったのか分からない。** `setup` や `reset` は破壊的なので、
   これは事故になる。

そこで `.haj` を**プロジェクト境界**として扱う。カレントから遡って最初に見つけた `.haj` が
そのプロジェクトであり、**そこで探索を打ち切る**。

### 2.2 `.haj/project`

境界と名前は `.haj/project` で宣言する。**このファイルは無くてもよい**(既定値で成立する)。
`key = value` を並べただけの形式で、`#` から行末はコメント。

```
name = example-app
root = true
```

| 鍵 | 既定 | 意味 |
|---|---|---|
| `name` | `.haj` を含むディレクトリの名前 | 表示名。一覧の出自ラベルと `HAJ_PROJECT` に使う |
| `root` | `true` | `true` ならここで遡上を止める(境界)。`false` なら親の `.haj` も探しに行く |

**`root = false` はモノレポのサブプロジェクト用。** 親の共通コマンド(`mig` など)も
継承したい場合にだけ、明示的に壁を開ける。

既定が `true`(=止まる)なので、**知らないうちに上流のコマンドが効くことはない**。
継承は常にオプトイン。

### 2.3 これがサブセットの仕組み

「そのリポジトリに置いたコマンドだけが、そのリポジトリで生える」。
プロジェクトごとに使えるコマンドが違うことは、**有効化リストのような別管理ではなく、
探索結果そのもの**として表現される。無効化したければファイルを置かなければよい。

同名のコマンドがあれば、より手前(=より具体的)が勝つ。プロジェクトの `build` が
全社共通の `build` を上書きする。

### 2.4 素性は常に見えていなければならない

同じ `haj setup` がプロジェクトごとに違う挙動をする以上、**いまどれが効いているのかが
見えないことは、それ自体が欠陥**である。コアは次の3つでこれを担保する。

1. **`haj` は現在のプロジェクトを先頭に出す。**

   ```
    プロジェクト: example-app  (/home/kurari/repos/.../example-app)
   ```

2. **一覧はコマンドごとに出自を出す。**

   ```
    hajコマンド (haj help <名前> で詳細):
      bao-login  Vaultにログイン              [共通]
      setup      セットアップ                  [example-app]
   ```

   出自ラベルは `[<プロジェクト名>]` / `[個人]` / `[共通]` / `[PATH]`。

3. **`haj which --all <名前>` は隠れている定義まで並べる。**

   ```
   $ haj which --all setup
   * /home/kurari/repos/example-app/.haj/commands/setup  [example-app]
     /usr/local/lib/haj/commands/setup                          [共通]

   (* が実行されるもの。他は隠れている)
   ```

### 2.5 「実行可能ファイル」の判定

- 通常ファイルであること(ディレクトリは無視する)
- 実行ビットが立っていること(`mode & 0o111 != 0`)
- シンボリックリンクは辿る。リンク切れは無視する

`commands/` に README や `.gitkeep` を置いても、実行ビットが無ければコマンドにはならない。

### 2.6 名前の制約

以下はサブコマンド名として認められない(置かれても無視される)。

- 空文字列
- `.` または `-` で始まるもの
- `/` を含むもの
- **予約語**: `help`, `commands`, `which`, `config`, `exec`, `selfupgrade`, `secrets`, `__complete`

予約語を弾くのは、`.haj/commands/help` を置かれるとコアのヘルプが奪われ、
「コマンド一覧を出す手段が無くなる」状態に陥りうるため。`exec` はさらに強い理由で
奪われてはならない — 置き換えられると**展開済みシークレットを横取りする最短経路**になる(§9.2)。

## 3. 実行

コアは **`exec(2)` で自分自身を置き換える**。ラッパープロセスを残さない。したがって:

- サブコマンドの終了コードはそのまま呼び出し元へ伝わる
- シグナル(Ctrl-C など)はサブコマンドが直接受け取る
- `ps` に余計なプロセスが並ばない

### 3.1 コアが渡す環境変数

| 変数 | 値 |
|---|---|
| `HAJ_ROOT` | そのサブコマンドが属するツリー(`commands/` の親)の絶対パス |
| `HAJ_NAME` | 呼ばれた名前(`haj mig` なら `mig`) |
| `HAJ_PROJECT` | いま**操作対象になっている**プロジェクトの名前(§2.2) |
| `HAJ_PROJECT_DIR` | そのプロジェクトのディレクトリ(`.haj` を含む方) |

`HAJ_ROOT` は**共通ライブラリを解決するためにある**。サブコマンドは

```sh
. "$HAJ_ROOT/lib/common.sh"
```

と書けばよい。プロジェクト固有のコマンドは**そのプロジェクトの lib** を読むことになり、
全社共通の lib と混線しない。

`HAJ_PROJECT` は**破壊的なコマンドが対象を名乗るためにある**。`setup` や `reset` は、
何に対して実行しているのかを出力に出すべきである。

```sh
echo "==> ${HAJ_PROJECT}: セットアップします (${HAJ_PROJECT_DIR})"
```

**`HAJ_ROOT` と `HAJ_PROJECT` は別物である。** 前者は「そのコマンドがどこから来たか」、
後者は「いまどこに対して実行しているか」。全社共通の `mig` をプロジェクトの中で叩けば、
`HAJ_ROOT=/usr/local/lib/haj` だが `HAJ_PROJECT=example-app` になる。
`root = false` の入れ子でも両者はずれる。

`$PATH` 上の `haj-<名前>` で見つかった場合はツリーが無いので、`HAJ_ROOT` は
**設定されない**(呼び出し元の環境に残っていても明示的に削除される)。同様に、
プロジェクトの外で実行した場合は `HAJ_PROJECT` も設定されない。

### 3.2 コアが返す終了コード

| コード | 意味 |
|---|---|
| サブコマンドの値 | 正常に exec できた場合(コアは何も足さない) |
| `127` | 未知のコマンド(シェルの command not found に合わせる) |
| `126` | 見つかったが exec に失敗した |
| `1` | コア自身の使い方の誤り(`haj which` に引数が無い等) |

## 4. コマンド規約

サブコマンドは、以下の引数で呼ばれたときの応答を実装する。

### 4.1 `--haj-describe` (必須)

**一行**の説明を stdout に出し、`0` で終了する。

```sh
$ haj-mig --haj-describe
DBマイグレーション (status/up/down/create/edit)
```

`haj help` のコマンド一覧はこれを全コマンドに聞いて**自動生成**される。
複数行返しても 1 行目だけが使われる。

これを実装しなくても**コマンドとしては動く**(一覧に説明が出ないだけ)。ただし
一覧の価値が落ちるので必須とする。

### 4.2 `--haj-help` (任意)

複数行の詳しい使い方を stdout に出し、`0` で終了する。`haj help <名前>` で表示される。

実装していない場合は `--haj-describe` の出力にフォールバックする。

### 4.3 `--haj-complete <語...>` (任意)

補完候補を**改行区切り**で stdout に出し、`0` で終了する。

`<語...>` は **そのサブコマンド以降に入力済みの語**。カーソル位置の語は含まない。

**語数の規則(最も間違えやすい)**

| ユーザーの入力 | 渡される語 | 返すべきもの |
|---|---|---|
| `haj mig ⇥` | (なし) | 1 語目の候補 |
| `haj mig up ⇥` | `up` | 2 語目の候補 |
| `haj mig create foo ⇥` | `create` `foo` | 3 語目の候補 |

つまり **`$#` が 0 なら 1 語目を補完する**。`[ $# -le 1 ]` と書くと 1 つずれる。

候補を列挙できない箇所(自由入力)では、丸括弧で囲んだ 1 行を返す。シェル補完は
これを候補ではなく**説明文**として表示する。

```
(新しいマイグレーションの名前)
```

### 4.4 規約フックの制約

- **stdin は `/dev/null`** に接続される。フックで入力を待ってはならない
- **stderr は捨てられる**
- **タイムアウトは既定 2 秒**(`HAJ_HOOK_TIMEOUT_MS` で変更可)。超えると `SIGKILL` され、
  そのコマンドは「説明なし/候補なし」として扱われる。他のコマンドは巻き添えにならない
- **副作用を持ってはならない。** フックは `haj help` と TAB のたびに呼ばれる

### 4.5 実装の型

規約フックは**共通ライブラリを読む前に**処理すること。説明文を 1 行返すためだけに
DB の環境変数を読み込むのは無駄で、TAB の体感速度に直結する。

```sh
#!/bin/bash
set -euo pipefail

case "${1:-}" in
  --haj-describe) echo "一行説明"; exit 0 ;;
  --haj-help)     cat <<'EOF'
詳しい使い方
EOF
    exit 0 ;;
  --haj-complete)
    shift
    if [ $# -eq 0 ]; then printf '%s\n' up down; else printf '%s\n' v0 v2; fi
    exit 0 ;;
esac

# ここから先が本体。重い初期化はフックの後に置く。
. "${HAJ_ROOT:?hajルーター経由で実行してください}/lib/common.sh"
...
```

## 5. ヘルプの構成

`haj help` の出力は次の 3 つを連結したもの。

```
help.header      ツリー直下のファイル。あれば先頭に出す(任意)
   ↓
コマンド一覧      --haj-describe を全コマンドに聞いて自動生成
   ↓
help.footer      ツリー直下のファイル。あれば末尾に出す(任意)
```

`help.header` / `help.footer` は**探索順で最初に見つかったツリー**のものが使われる。
プロジェクトが自分の header を置けば、全社共通の header を上書きできる。

ここに置くのは「コマンド以外の案内」だけ(URL、環境変数の上書き方法、ファイルの
置き場所など)。**コマンド一覧は自動生成されるので、手で書いてはならない。**

---

## 6. 補完プロトコル

シェル補完スクリプトは候補を一切持たない。コアに聞くだけ。

```
haj __complete                    → "名前\t説明" を1行ずつ
haj __complete <名前> [語...]       → そのコマンドの --haj-complete へ転送
```

したがって**サブコマンドを追加しても補完スクリプトを更新する必要がない**。
これが `--haj-complete` を規約に含める理由。

---

## 7. 説明文のキャッシュ

`haj help` と TAB 補完はコマンドの数だけサブプロセスを起動する。bash のサブコマンド
1 本あたり実測で 9ms 程度かかるため、20 本あれば TAB のたびに約 200ms 待たされる。

コアは `--haj-describe` の結果を **(パス, 更新時刻, サイズ)** を鍵にキャッシュする。

- 置き場所: `$XDG_CACHE_HOME/haj/describe.tsv`(既定 `~/.cache/haj/describe.tsv`)
- ファイルを触っていないコマンドは二度と起動されない
- `HAJ_NO_CACHE=1` で無効化できる
- 書き込みは一時ファイル + `rename` で行う(補完は同時に何本も走りうるため)

キャッシュは**壊れても消えても正しく動く**(聞き直すだけ)。

## 8. 設定

### 8.1 場所(XDG に従う)

**git と同じ形**。リポジトリ側は `.haj/`(git の `.git/`)、ユーザー側は
`~/.config/haj/`(git の `~/.config/git/config`)。

| 何 | 場所 |
|---|---|
| ユーザー設定 | `$XDG_CONFIG_HOME/haj/config`(既定 `~/.config/haj/config`) |
| 個人用コマンド | `$XDG_CONFIG_HOME/haj/commands/` |
| 説明文キャッシュ | `$XDG_CACHE_HOME/haj/describe.tsv`(既定 `~/.cache/haj/`) |
| プロジェクト設定 | `<リポジトリ>/.haj/project` |

0.3.0 までの `~/.haj/commands/` も当面は読む(置いてあるコマンドが黙って消えるのが
一番たちが悪い)。新規は `~/.config/haj/commands/` に置く。

### 8.2 形式

**`.haj/project` と同じ `key = value`。** `#` から行末はコメント。

設定ファイルの形式が2つあると「どっちがどっちだったか」を覚える羽目になる。
入れ子が要るような項目は無く、信頼済みツリーの一覧のような**列**は direnv 方式で
別ファイルにすればよい。これで依存クレートをゼロに保てる(YAML/TOML はパーサが要る)。

```
# ~/.config/haj/config
gitlab     = https://gitlab.avaper.day
project_id = 788
target     = x86_64-unknown-linux-musl
token      = glpat-xxxxxxxx

hook_timeout_ms = 2000
```

### 8.3 優先順位

**環境変数 > 設定ファイル > 既定値。**

この3段が見えないと「なぜ効かないのか」を調べる手段が無くなるので、
**`haj config` は実効値と一緒に必ず出所を出す**(`haj which` が探索順を見せるのと
同じ理由)。`token` は値を出さず、設定されているかと出所だけを示す。

| 設定ファイルの鍵 | 環境変数 | 既定 | 意味 |
|---|---|---|---|
| `command_path` | `HAJ_COMMAND_PATH` | `/usr/local/lib/haj/commands` | システム共通のコマンド置き場(`:` 区切り) |
| `hook_timeout_ms` | `HAJ_HOOK_TIMEOUT_MS` | `2000` | 規約フックのタイムアウト |
| `op_cmd` | `HAJ_OP_CMD` | `op` | op 参照の解決に使う CLI(§10) |
| `vault_cmd` | `HAJ_VAULT_CMD` | `bao` | vault 参照の解決に使う CLI(§10) |
| `vault_addr` | `VAULT_ADDR` | `https://vault.avap.plus` | vault サーバ。環境の `VAULT_ADDR` / `BAO_ADDR` が優先(§10.4) |
| `vault_login` | `HAJ_VAULT_LOGIN` | `-method=oidc -path=id-avap-keycloak` | 未ログイン時に自動実行する `login` の引数(§10.4)。`off` で無効化 |
| `token` | `HAJ_TOKEN` | (無し) | `selfupgrade` が使う GitLab トークン |
| `gitlab` | `HAJ_GITLAB` | `https://gitlab.avaper.day` | GitLab インスタンス |
| `project_id` | `HAJ_PROJECT_ID` | `788` | haj のプロジェクト ID |
| `target` | `HAJ_TARGET` | `x86_64-unknown-linux-musl` | 取得するビルドのターゲット |

環境変数だけのもの(設定ファイルに書けない):

| 変数 | 意味 |
|---|---|
| `HAJ_NO_CACHE` | `1` で説明文キャッシュを無効化(デバッグ用) |
| `XDG_CONFIG_HOME` / `XDG_CACHE_HOME` | 置き場所そのものを決めるので、設定ファイルには書けない |
| `HAJ_SECRETS` | `1` でシークレット参照の展開を有効化(§10)。金庫を開ける鍵を設定ファイルに常設させない |

## 9. コア組み込みのコマンド

| コマンド | 用途 |
|---|---|
| `haj` / `haj help` | ヘルプ(header + 自動生成の一覧 + footer) |
| `haj help <名前>` | そのコマンドの `--haj-help` |
| `haj commands` | `名前\tパス\t出自\t説明` を機械可読で列挙 |
| `haj which [--all] <名前>` | 探索で勝っている実行ファイルのパス(`--all` で隠れているものも) |
| `haj config` | 設定の実効値と、その出所(§8) |
| `haj selfupgrade` | コア自身の更新(§9.1) |
| `haj secrets` | シークレット参照の展開対象を確認する(§10.6) |
| `haj exec <プログラム> [引数...]` | PATH のコマンドにシークレットを注入して実行(§9.2) |
| `haj --version` | コアの版 |
| `haj __complete ...` | 補完プロトコル(人間向けではない) |

### 9.0 組み込みは常に見えていなければならない

**組み込みコマンドは探索の対象ではないが、`haj help` の一覧にも補完にも `haj commands` にも
出す。** どこにいても使えるものを一覧から漏らせば、「`haj help` の一覧が実態と一致する」
という haj の約束そのものが嘘になる。

一覧では、プロジェクトのコマンドとは節を分けて出す(性質が違うため)。

```
 hajコマンド (haj help <名前> で詳細):
   mig          DBマイグレーション   [example-app]
   bao-login    Vaultにログイン      [共通]

 haj自身 (どのプロジェクトでも使える):
   help         使い方を表示する (haj help <名前> で個別)
   commands     コマンド一覧を機械可読で出す
   which        どの定義が効いているかを見る (--all で隠れているものも)
   selfupgrade  haj自身を更新する
   secrets      シークレット参照の展開対象を確認する (dry-run)
   exec         PATHのコマンドにシークレットを注入して実行する
```

`haj commands` では出自ラベル `[haj]`、パスの代わりに `(組み込み)` を出す。

組み込みも `haj help <名前>` に答え、補完候補を返す(`haj selfupgrade ⇥` → `--check`、
`haj which ⇥` → `--all` とコマンド名)。実装がプロセス起動ではないだけで、**サブコマンドと
同じ顔をしている**。

### 9.1 `haj selfupgrade`

コア自身をPackage Registryのリリースで置き換える。install.sh と同じ経路・同じ
環境変数を使い、チェックアウト無しで「入っているhajを更新する」ためのもの。

```
haj selfupgrade [<版>] [--check]
```

- 引数なし: 最新リリースを調べ、今の版と**違えば**置き換える。同じなら何もしない
- `<版>` 指定: その版を**無条件に**入れる。同版の再インストールもダウングレードも
  これで表現する(`--force` は設けない)
- `--check`: 調べるだけ。終了コードで返す — `0` 最新 / `1` 更新あり / `2` 調べられなかった
- 版の比較は**文字列の一致のみ**。semver は解釈しない(違えば更新とみなす)

**取得と検証の手順**

1. 版が未指定なら releases API の先頭から `tag_name` を取る(先頭の `v` は落とす)
2. `haj-$HAJ_TARGET.tar.gz` を Package Registry から取得する
3. `.sha256` が取れたら `sha256sum -c` で照合する。**不一致は中止**。
   `.sha256` 自体が無ければ照合は省略する(install.sh と同じ)
4. 展開した新バイナリを `--version` で起動し、出力が `haj ` で始まることを確認する。
   壊れたバイナリで自分を潰さないための健全性チェック
5. 現バイナリと**同じディレクトリ**に一時ファイル(mode 0755)を書き、`rename(2)` で
   置き換える。同一ファイルシステム内なので原子的で、実行中のプロセスに影響しない
6. 置き場所に書けなければ、sudo での再実行を**提案して**終了する。コアは昇格しない

**依存ゼロとの折り合い**: stdlib に HTTP は無いが crate は増やさない。取得・展開・照合は
`curl` / `tar` / `sha256sum` を子プロセスとして呼ぶ(exec がコアの本業、という設計に沿う)。
`curl` が無い環境ではその旨を伝えて終了する。

**読む環境変数**(install.sh と共通)

| 変数 | 既定 | 意味 |
|---|---|---|
| `HAJ_GITLAB` | `https://gitlab.avaper.day` | GitLabインスタンス |
| `HAJ_PROJECT_ID` | `788` | hajのプロジェクトID |
| `HAJ_TARGET` | `x86_64-unknown-linux-musl` | 取得するビルドのターゲット |
| `HAJ_TOKEN` | (必須) | 取得に使うトークン。`CI_JOB_TOKEN` と同値なら JOB-TOKEN ヘッダを使う |

### 9.2 `haj exec`

探索を**通さず**、PATH のコマンド(`/` を含めばそのパス)に**シークレットの注入だけ
して**実行する。`op run` / `doppler run` が占めている場所。「注入は欲しいが、haj の
コマンドにするほどではない」一回きりの実行のためにある。

```sh
haj --secret DB_HOST=vault://avap/data/db/host exec sh -c 'mysql -h $DB_HOST'
```

- 展開の規則(§10)はサブコマンド実行と完全に同じ。受け渡しフラグ(§10.7)も
  ambient な走査(`HAJ_SECRETS=1`)も同様に乗る
- **シェルの変数展開が要るなら、明示的に `sh -c` を書く。** コアは文字列をシェルに
  包まない(§11 — コアは引数を解釈しない)。`$VAR` を使わないなら
  `haj exec mysql -h db.example` のようにそのまま並べればよい
- `HAJ_ROOT` / `HAJ_NAME` / `HAJ_PROJECT` / `HAJ_PROJECT_DIR` は**渡さない**(明示的に
  削除する)。haj の外の世界のコマンドに、haj の顔をさせない
- 見つからなければ `127`、exec に失敗したら `126`(§3.2 と同じ)
- `exec` は予約語(§2.6)。`.haj/commands/exec` で置き換えられると、`--secret` で
  展開したシークレットを横取りする最短経路になるため、**探索より常に優先**する

## 10. シークレット参照(展開)

サブコマンドに渡す値には、シークレットの実体ではなく**参照**を書ける。コアは exec の
直前に参照を解決し、展開済みの値を子プロセスに渡す。リポジトリにも環境にも平文を置かず、
haj がプロキシになる。

**参照の書式は発明しない。** 1Password は `op inject` の書式、Vault は vault-agent の
template で使う展開式を、それぞれ**そのまま**受け付ける。既存のテンプレートや
ドキュメントからコピペで移せることを設計要件とする。

### 10.1 有効化(オプトイン)

展開は **`HAJ_SECRETS=1` のときだけ**行われる。既定では何もしない(参照はただの文字列
としてそのまま渡る)。clone した直後のリポジトリで勝手に金庫が開く、という事故を防ぐため、
展開は常に明示のオプトイン。CI などの管理された環境ではジョブの環境変数に置けばよい。

プロジェクトのツリー由来の値(`.haj/env`)には、これに**加えて**「許可したツリーだけ」の
ゲートがかかる(env 注入の仕組みと一緒に入る。まだ無い)。

### 10.2 展開の対象

ソースは3層あり、それぞれゲートが違う。

| 層 | 指定 | ゲート |
|---|---|---|
| その1回だけ | `--secret` / `--env` / `--secretfile`(§10.7) | フラグを打ったこと自体が同意 |
| ユーザーの環境 | 環境変数の値の走査と、argv | `HAJ_SECRETS=1` |
| プロジェクト | `.haj/env` 等の専用ファイル(未実装) | ツリー信頼 + オプトイン |

規約フック(`--haj-describe` / `--haj-complete` / `--haj-help`)には**展開しない**。
TAB を押すたびに金庫へ問い合わせが飛ぶのは論外(タッチ認証で固まる)。展開は
「本体を実行する」ときだけ。

### 10.3 参照の書式

| 参照 | 意味 |
|---|---|
| `op://<金庫>/<アイテム>/[<セクション>/]<フィールド>` | 1Password。**`op inject` に丸ごと委譲**し、書式も意味論も inject に従う |
| `{{ with secret "<パス>" }}{{ .Data.data.<フィールド> }}{{ end }}` | Vault。vault-agent template の正準形。KV v2 の `/data/` 入りパスをそのまま書く |
| `vault://<パス>/<フィールド>` | 上の短縮形。**最後のセグメントがフィールド**、残りがパス(パスの規約は template 形と同じ) |
| `env://VAR` | 環境変数 VAR の値。再帰はしない(1段だけ) |
| `file://<パス>` | ファイルの中身。docker secrets / systemd credentials との接続に |

**環境変数の走査は、値全体が参照のときだけ**展開する(`op://` / `vault://` 等で始まる、
または `{{` で始まり `}}` で終わる)。文字列中への埋め込みは解釈しない。接続文字列の
組み立てはサブコマンド側の責務。

埋め込みを許すと「参照をたまたま文中に含む」だけの変数まで解決しに行ってしまう。
実例: GitLab の MR パイプラインは `CI_MERGE_REQUEST_DESCRIPTION` に MR の説明文を
入れる。説明文に `op://` の例が書いてあるだけで、op の解決が走って全体が止まる。

**コマンドライン引数では op の埋め込みも展開する**(inject の意味論のまま)。
argv は人がその実行のために明示的に書いたものなので、誤爆の心配がない。

```
DB_PASSWORD=vault://avap/data/hoge/fuga                               → 展開される
OTP={{ with secret "avap/data/hoge" }}{{ .Data.data.fuga }}{{ end }}  → 展開される(上と等価)
TOKEN=op://Infra/ci/token                                             → op inject が展開
DB_URL=postgres://u:vault://a/b/c@host/db                             → 展開されない(ただの文字列)
NOTE="説明: op://a/b/c を参照…"                                        → 展開されない(値全体が参照ではない)

haj deploy --token "Bearer op://Infra/ci/token"                       → 引数なので埋め込みも展開される
```

vault の template 形は正準形**のみ**解釈する。`printf` 等を含む任意の式が来たら
「正準形のみ対応」と言って中止する。

### 10.4 解決

依存クレートは増やさない。op / vault は CLI を子プロセスとして呼び、env / file は
stdlib だけで解決する。

| 参照 | 解決方法 |
|---|---|
| op | `op inject` に値を stdin で渡し、stdout を採る |
| vault | `vault kv get -field=<フィールド> [-mount=<マウント>] <パス>`。パスの2セグメント目が `data` なら KV v2 の API パスとみなし、`-mount=<先頭>` と相対パスに読み替える |

CLI は差し替えられる(§8.3): `HAJ_OP_CMD` / 設定 `op_cmd`(既定 `op`)、
`HAJ_VAULT_CMD` / 設定 `vault_cmd`(既定 `bao`)。

**vault サーバ**: 環境に `VAULT_ADDR` / `BAO_ADDR` があればそれを尊重する。無ければ
設定 `vault_addr`(既定 `https://vault.avap.plus`)を両方の名前で CLI に渡す
(bao は `BAO_ADDR` を先に見る)。

**vault の自動ログイン**: 未ログインのとき(`token lookup` が非 0)に限り、解決の前に
`login` を**端末を継いで**実行する。引数は設定 `vault_login`(環境変数
`HAJ_VAULT_LOGIN`)から取り、既定は `-method=oidc -path=id-avap-keycloak`。

```
# ~/.config/haj/config — 既定から変えたいときだけ書く
vault_login = -method=oidc -path=id-avap-keycloak role=direct callbackmode=direct
```

- `vault_login = off` で無効化する。そのときは解決が vault 自身のエラーで
  fail-fast する(従来どおり)
- ログイン状態の確認は 1 プロセスにつき 1 回だけ。ログイン済みなら何も起きない
- 引数は空白区切りで分割する(引用符やエスケープは解釈しない)
- **認証しない CI で vault 参照を使うなら `HAJ_VAULT_LOGIN=off` を置くこと。**
  OIDC ログインはブラウザと人を待つ。`VAULT_TOKEN` 等で認証済みの CI では
  `token lookup` が通るので login は走らない

- **タイムアウトは設けない。** op のタッチ認証など、人を待つ場面が正当にある。
  規約フックの 2 秒とは別物。
- 展開結果の末尾の改行 1 つは落とす(CLI や credential ファイルが付けるもの)。
- **キャッシュしない。** 毎回聞く。セッション管理は各 CLI の仕事。

### 10.5 失敗したら止まる

解決に失敗した(CLI が無い・非 0 で終わった・参照の書式が壊れている・変数やファイルが
無い)ら、**本体を実行せずに exit 1**。未解決の参照文字列が DB パスワードとして
そのまま使われ、分かりにくく壊れる事故を防ぐ(fail-fast)。

### 10.6 `haj secrets`(dry-run)

何が展開対象なのかを、**解決せずに**確かめる。

```
$ haj secrets
HAJ_SECRETS=1 (展開は有効)

 環境変数:
   DB_PASSWORD  vault://avap/data/hoge/fuga
   OP_TOKEN     op://Infra/ci/token
```

参照の**対象**(パス)は出すが、**値**は解決しない。金庫に問い合わせない。

### 10.7 明示的な受け渡し(`--secret` / `--env` / `--secretfile`)

環境の走査(§10.2 の2層目)は「値が参照かどうか」しか見ておらず、「**誰に**渡すか」を
見ていない。DB パスワードを必要としないコマンドにも実体が渡るし、信頼していない
ツリーのコマンドにも届いてしまう。**渡すものと相手を、人がその実行時に明示する**のが
この3フラグ。

```sh
haj --secret DB_HOST=vault://avap/data/db/host \
    --secret DB_PASS=vault://avap/data/db/password \
    --secretfile config.ini=config.ini.tpl \
    --env ./mig.env \
    mig up
```

- **フラグはサブコマンド名の前にだけ書ける**(git 方式)。名前以降は §11 のとおり
  無解釈で素通しなので、`haj mig --secret ...` の `--secret` は mig にそのまま渡る
- **フラグを打ったこと自体が同意。`HAJ_SECRETS=1` は要らない**
- 参照は argv に見えるが、参照は秘密ではない。**実体は環境変数かファイルに置かれ、
  argv には乗らない**
- 同じ変数への指定が重なったら、**書いた順で後勝ち**
- フラグの後にコマンド名が無い、または組み込みコマンドが続く場合は使い方の誤り(exit 1)

**`--secret <名前>=<値>`**(複数可) — 値を展開して環境変数 <名前> で渡す。
明示なので op の埋め込みも展開する(argv と同じ規則)。参照でなければ**平文として
そのまま**渡す。

**`--env <ファイル>`**(複数可) — `key = value` ファイル(`.haj/project` と同じ形式)を
読み、各値を**値全体規則**(環境変数の走査と同じ)で展開して渡す。参照だけ書いた
ファイルをリポジトリにコミットし、使うときに明示指定する、という中間層。

**`--secretfile <出力>=<テンプレート>`**(複数可) — テンプレートを描画し、<出力> に
**mode 0600** で書いてから exec する。環境変数に乗らない設定ファイル派のツール向け。

- 描画規則: vault は正準形ブロック `{{ with secret "<パス>" }}{{ .Data.data.<フィールド> }}{{ end }}`
  の置換、op はその後にファイル全体を `op inject` へ(inject の意味論)。
  `vault://` 短縮形はテンプレート内では解釈しない(トークンの境界が曖昧になる)
- **全て解決できてから書く**。失敗時は出力を書かず fail-fast(半端なファイルを残さない)
- **描画したファイルを haj は消さない**。コアは exec(2) で自分を置き換えるため、
  実行後の後始末は構造的に不可能。残ることを前提に、消すのは使う側の責任

### 10.8 注意(仕様ではなく性質)

- argv に展開したシークレットは `ps` / `/proc/*/cmdline` から見える。**シークレットは
  環境変数で渡すこと**。argv 展開は「参照を引数で受けたい」場面のためにあるが、
  実体を argv に晒す危険は使う側が負う。
- 展開後の値は子プロセスの `/proc/<pid>/environ` に見える(環境変数渡しの本質的な限界)。

## 11. 意図的にやらないこと

- **タスクを宣言型ファイル(`haj.toml` の `run = "..."`)で書けるようにしない。**
  分岐や冪等性判定を含む現実のタスクは宣言型に収まらず、「シェルスクリプトを
  TOML の文字列に埋め込んだもの」に堕する。タスクはファイルとして書く。
- **コアがサブコマンドの引数を解釈しない。** 引数はそのまま渡す。コアが知るのは
  「名前」だけ。
- **サブコマンドの実装言語を制約しない。** 契約は exec と引数と stdout だけで定義され、
  言語に依存しない。これによりコアを将来別の言語で書き直しても、サブコマンドは
  1 行も直さずに動く。
